Mathematical Foundations of Halo2 and Zero-Knowledge Machine Learning

Prerequisites. This document assumes the following background:
  • Modular arithmetic — comfortable with $a \bmod p$, greatest common divisors, and basic number theory (covered in any introductory discrete mathematics course).
  • Linear algebra — vectors, matrices, dot products; used in the ML sections.
  • Polynomials — polynomial division, roots, and the factor theorem at the high-school level.
  • Probability — basic probability notation $\Pr[\cdot]$; no measure theory needed.
  • Programming familiarity — pseudocode and Rust-like syntax appear in examples; no deep Rust knowledge required.
Not required: prior knowledge of cryptography, elliptic curves, or zero-knowledge proofs — these are developed from first principles in Sections 1–4.

This project implements ZK Machine Learning (ZKML): running ML algorithms (linear regression, logistic regression, decision trees) inside zero-knowledge circuits. The core challenge is that ZK circuits over the BN254 field natively support only $+$, $-$, $\times$ on unsigned integers—no fractions, no floats, no transcendental functions. This document builds the mathematical foundations needed to bridge that gap, from finite fields up through the Halo2 proving system to fixed-point ML circuits.

1. Finite Fields

All computation inside a zero-knowledge proof happens over a finite field, not ordinary real numbers. Think of it as doing arithmetic on a clock face, where the numbers wrap around.

Definition 1.1 — Finite Field $\mathbb{F}_p$ Let $p$ be a prime number. The set $\mathbb{F}_p = \{0, 1, 2, \dots, p-1\}$ equipped with addition and multiplication modulo $p$ forms a finite field. That is, for any $a, b \in \mathbb{F}_p$: $$a +_p b \;=\; (a + b) \bmod p, \qquad a \cdot_p b \;=\; (a \cdot b) \bmod p.$$

The key properties that make $\mathbb{F}_p$ a field (analogous to the reals $\mathbb{R}$, but finite):

  • Closure: addition and multiplication stay inside $\{0,\dots,p-1\}$.
  • Inverses: every nonzero $a$ has a unique multiplicative inverse $a^{-1}$ such that $a \cdot a^{-1} \equiv 1 \pmod{p}$. (Computed via Fermat's little theorem: $a^{-1} = a^{p-2} \bmod p$.)
  • No zero divisors: if $a \cdot b \equiv 0 \pmod p$, then $a = 0$ or $b = 0$.
Example In $\mathbb{F}_7$: $\;5 + 4 = 9 \bmod 7 = 2$, $\;3 \times 5 = 15 \bmod 7 = 1$, so $3^{-1} = 5$.

This project uses the BN254 scalar field, where:

$$p = 21888242871839275222246405745257275088548364400416034343698204186575808495617$$

This is a 254-bit prime. Every value we compute (weights, inputs, polynomial coefficients) lives in $\mathbb{F}_p$.

1.1 Cyclic Groups

A group $(G, \circ)$ is a set $G$ with an operation $\circ$ that is associative, has an identity, and every element has an inverse.

Definition 1.2 — Cyclic Group A group $G$ is cyclic if there exists a generator $g \in G$ such that every element of $G$ can be written as $g^k$ for some integer $k$. We write $G = \langle g \rangle$. The order of $G$, written $|G|$, is the number of elements in $G$. For example, $|\mathbb{F}_7^*| = |\{1,2,3,4,5,6\}| = 6$.

The multiplicative group $\mathbb{F}_p^* = \mathbb{F}_p \setminus \{0\}$ is cyclic of order $p-1$. This means there exists a generator $g$ such that $\{g^0, g^1, g^2, \dots, g^{p-2}\} = \mathbb{F}_p^*$.

Crucially, if $n$ divides $p-1$ (written $n \mid (p-1)$), then there exists an element $\omega \in \mathbb{F}_p$ of order $n$, meaning $\omega^n = 1$ and $\omega^k \neq 1$ for $0 < k < n$. These are the $n$-th roots of unity, and they enable the NTT (Number Theoretic Transform) that makes polynomial arithmetic efficient inside Halo2.

Example — Roots of Unity In $\mathbb{F}_7$, we have $p - 1 = 6$. Since $3 \mid 6$, there exists an element of order 3. Take $\omega = 2$: we get $2^1 = 2,\; 2^2 = 4,\; 2^3 = 8 \equiv 1 \pmod 7$. So $\{1, 2, 4\}$ are the 3rd roots of unity in $\mathbb{F}_7$. Halo2 uses this same idea with $n = 2^k$ (a power of two) to lay out the execution trace and perform fast polynomial evaluation.

1.2 Extension Fields

Just as the real numbers $\mathbb{R}$ can be extended to the complex numbers $\mathbb{C} = \mathbb{R}[i]/(i^2+1)$ by adjoining $\sqrt{-1}$, finite fields can be extended similarly.

Definition 1.3 — Extension Field $\mathbb{F}_{p^k}$ Let $m(x)$ be an irreducible polynomial of degree $k$ over $\mathbb{F}_p$ (meaning it cannot be factored into lower-degree polynomials). The extension field $\mathbb{F}_{p^k}$ consists of all polynomials of degree $< k$ with coefficients in $\mathbb{F}_p$, where arithmetic is performed modulo $m(x)$: $$\mathbb{F}_{p^k} \;=\; \mathbb{F}_p[x]\,/\,\bigl(m(x)\bigr) \;=\; \{a_0 + a_1 x + \cdots + a_{k-1}x^{k-1} : a_i \in \mathbb{F}_p\}.$$ This field has $p^k$ elements. Addition is component-wise; multiplication is polynomial multiplication reduced modulo $m(x)$.
Example — $\mathbb{F}_{5^2}$

Take $p = 5$ and the irreducible polynomial $m(x) = x^2 + 2$ over $\mathbb{F}_5$. (It's irreducible because neither $0, 1, 2, 3, 4$ is a root: $0^2 + 2 = 2$, $1^2+2=3$, $2^2+2=1$, $3^2+2=1$, $4^2+2=3$—none are zero mod 5.)

Elements look like $a + bx$ with $a, b \in \mathbb{F}_5$. There are $5^2 = 25$ elements. To multiply $(2 + 3x)(1 + 4x)$:

$$2 + 8x + 3x + 12x^2 = 2 + 11x + 12x^2 \equiv 2 + x + 2x^2 \pmod{5}.$$

Now reduce $x^2 \equiv -2 \equiv 3 \pmod{m(x)}$, giving $2 + x + 2 \cdot 3 = 8 + x \equiv 3 + x \pmod{5}$.

Extension fields matter for elliptic curve pairings (Section 3). The pairing's target group $\mathbb{G}_T$ lives in $\mathbb{F}_{p^{12}}^*$—a degree-12 extension of $\mathbb{F}_p$. Each element of $\mathbb{F}_{p^{12}}$ is a polynomial of degree 11 with $\mathbb{F}_p$ coefficients, so it takes 12 field elements to represent. In practice, tower extensions like $\mathbb{F}_{p^2} \to \mathbb{F}_{p^6} \to \mathbb{F}_{p^{12}}$ are used for efficiency.

2. Elliptic Curves

Elliptic curves provide the "hard problems" that make cryptography secure. They also give us the algebraic structure needed for polynomial commitments.

2.1 The Curve Equation

Definition 2.1 — Elliptic Curve An elliptic curve over a field $\mathbb{F}$ is the set of points $(x, y) \in \mathbb{F}^2$ satisfying the short Weierstrass equation: $$E : \quad y^2 = x^3 + ax + b, \qquad 4a^3 + 27b^2 \neq 0,$$ together with a special point at infinity $\mathcal{O}$.

The condition $4a^3 + 27b^2 \neq 0$ ensures the curve has no singular points (cusps or self-intersections). For the BN254 curve used in this project, $a = 0$ and $b = 3$, giving:

$$E_{\text{BN254}} : \quad y^2 = x^3 + 3 \quad \text{over } \mathbb{F}_p.$$
Why "discrete"? When we draw elliptic curves in textbooks (and in the figure below), we plot them over the real numbers $\mathbb{R}$, which gives a smooth, continuous curve. But in cryptography, we work over a finite field $\mathbb{F}_p$. There, $x$ and $y$ can only take values in $\{0, 1, \dots, p-1\}$, so the "curve" is really a discrete set of scattered points—no lines, no tangents in the geometric sense. The algebraic formulas for point addition (slopes, intercepts) still work identically, because they only use field operations ($+, -, \times, \div$), which $\mathbb{F}_p$ supports. The smooth curve picture is just a visual aid; the actual computation happens on isolated dots in a finite grid.
Example — A tiny curve over $\mathbb{F}_{23}$ Consider $y^2 = x^3 + x + 1$ over $\mathbb{F}_{23}$. We check every $x \in \{0, \dots, 22\}$, compute $x^3 + x + 1 \bmod 23$, and see if the result is a perfect square mod 23. For instance, $x = 0$: $0 + 0 + 1 = 1$, and $1 = 1^2$, so $(0, 1)$ and $(0, 22)$ are on the curve. Repeating for all $x$ yields 28 points plus $\mathcal{O}$, giving $|E(\mathbb{F}_{23})| = 29$. These 29 points are spread seemingly at random across the $23 \times 23$ grid—nothing like a smooth curve.

But how do we determine whether a value has a square root in $\mathbb{F}_p$, and how do we compute it? This is where Euler's criterion and Fermat's little theorem come in.

Definition 2.1b — Quadratic Residue An element $a \in \mathbb{F}_p^*$ is a quadratic residue (QR) if there exists some $r \in \mathbb{F}_p$ such that $r^2 \equiv a \pmod{p}$. Otherwise it is a non-residue (NR). Exactly half of the nonzero elements of $\mathbb{F}_p$ are QRs.

Euler's criterion gives a simple test: $a$ is a QR if and only if

$$a^{(p-1)/2} \equiv 1 \pmod{p}.$$

If the result is $p - 1$ (i.e., $-1$), then $a$ has no square root in $\mathbb{F}_p$, and the point $(x, \cdot)$ is not on the curve.

When $a$ is a QR and $p \equiv 3 \pmod{4}$ (which is the common case for many cryptographic primes), the square root can be computed directly:

$$r = a^{(p+1)/4} \bmod p.$$
Why this works Verify: $r^2 = a^{(p+1)/2} = a^{(p-1)/2} \cdot a = 1 \cdot a = a \pmod{p}$, where the last step uses Euler's criterion ($a^{(p-1)/2} = 1$ for a QR). Since $y^2 = a$ is symmetric, the two roots are $r$ and $p - r$ (i.e., $-r$).
Example — Square root in $\mathbb{F}_{23}$

Is $x = 0$ on the curve $y^2 = x^3 + x + 1$ over $\mathbb{F}_{23}$? Compute $a = 0^3 + 0 + 1 = 1$.

  • Test (Euler): $1^{(23-1)/2} = 1^{11} = 1 \equiv 1 \pmod{23}$. Yes, 1 is a QR.
  • Square root: Since $23 \equiv 3 \pmod 4$, compute $r = 1^{(23+1)/4} = 1^6 = 1$.
  • Points: $(0,\; 1)$ and $(0,\; 23 - 1) = (0,\; 22)$.

Now try $x = 1$: $a = 1 + 1 + 1 = 3$. Test: $3^{11} \bmod 23 = 177147 \bmod 23 = 1$. So 3 is a QR. Root: $r = 3^{6} = 729 \bmod 23 = 16$. Points: $(1, 16)$ and $(1, 7)$.

Now try $x = 3$: $a = 27 + 3 + 1 = 31 \equiv 8 \pmod{23}$. Test: $8^{11} \bmod 23 = 22 \equiv -1$. So 8 is a NR—no points at $x = 3$.

For primes where $p \equiv 1 \pmod 4$ (including the BN254 prime), the Tonelli-Shanks algorithm is used instead. The details are more involved, but the key takeaway is: finding square roots in $\mathbb{F}_p$ is efficient (polynomial time), so enumerating curve points is computationally feasible.

2.2 The Group Law (Point Addition)

The remarkable fact is that the points on an elliptic curve form an abelian group under a geometric addition rule. The identity element is $\mathcal{O}$.

Case 1: Adding two distinct points $P + Q$

Given $P = (x_1, y_1)$ and $Q = (x_2, y_2)$ with $x_1 \neq x_2$:

  1. Draw the line through $P$ and $Q$. It intersects $E$ at exactly one more point $R'$.
  2. Reflect $R'$ over the $x$-axis to get $R = P + Q$.

The formulas are:

$$\lambda = \frac{y_2 - y_1}{x_2 - x_1}, \qquad x_3 = \lambda^2 - x_1 - x_2, \qquad y_3 = \lambda(x_1 - x_3) - y_1.$$

Case 2: Point doubling $P + P = 2P$

When $P = Q$, the "line through $P$ and $Q$" becomes the tangent line at $P$. Using implicit differentiation on $y^2 = x^3 + ax + b$:

$$\lambda = \frac{3x_1^2 + a}{2y_1}, \qquad x_3 = \lambda^2 - 2x_1, \qquad y_3 = \lambda(x_1 - x_3) - y_1.$$

Case 3: Inverse

The inverse of $P = (x, y)$ is $-P = (x, -y)$. If $P + Q = \mathcal{O}$, then $Q = -P$.

Why calculus matters here: The tangent-line formula for point doubling uses the derivative $dy/dx = (3x^2 + a)/(2y)$, obtained by implicitly differentiating the curve equation—the same technique from Calculus I.

2.3 Scalar Multiplication and the Discrete Logarithm

Scalar multiplication is defined as repeated addition: for an integer $k$,

$$[k]P = \underbrace{P + P + \cdots + P}_{k \text{ times}}.$$

This can be computed efficiently in $O(\log k)$ steps via double-and-add. However, the reverse problem (given $P$ and $Q = [k]P$, find $k$) is called the Elliptic Curve Discrete Logarithm Problem (ECDLP), and it is believed to be computationally intractable. This hardness is the foundation of elliptic-curve cryptography.

Definition 2.2 — Curve Group Order The set of points $E(\mathbb{F}_p) = \{(x,y) \in \mathbb{F}_p^2 : y^2 = x^3 + ax + b\} \cup \{\mathcal{O}\}$ forms a finite abelian group. Its order $n = |E(\mathbb{F}_p)|$ satisfies Hasse's bound: $|n - (p+1)| \leq 2\sqrt{p}$.

For BN254, the group has a large prime-order subgroup of order $r$, which is the scalar field we do arithmetic in.

3. Elliptic Curve Pairings

Pairings are a powerful tool that enables the verification step in KZG polynomial commitments. A pairing "multiplies" points from two different curve groups and maps them into a third group.

3.1 Bilinear Maps

Definition 3.1 — Bilinear Pairing A bilinear pairing is a map $$e : \mathbb{G}_1 \times \mathbb{G}_2 \to \mathbb{G}_T$$ where $\mathbb{G}_1, \mathbb{G}_2$ are (additive) elliptic curve groups of prime order $r$, and $\mathbb{G}_T$ is a (multiplicative) subgroup of $\mathbb{F}_{p^k}^*$ of the same order $r$, satisfying:
  1. Bilinearity: $e([a]P, [b]Q) = e(P, Q)^{ab}$ for all $a, b \in \mathbb{Z}$ and $P \in \mathbb{G}_1, Q \in \mathbb{G}_2$.
  2. Non-degeneracy: if $P \neq \mathcal{O}$ and $Q \neq \mathcal{O}$, then $e(P, Q) \neq 1$.
  3. Computability: $e$ can be computed efficiently (via the Tate or optimal Ate pairing).

The integer $k$ is called the embedding degree. For BN254, $k = 12$, meaning $\mathbb{G}_T \subset \mathbb{F}_{p^{12}}^*$.

3.2 Why Pairings Matter

The bilinearity property is the key. It lets a verifier check polynomial equations "in the exponent"—that is, verify relationships between secret values that are hidden inside elliptic curve points, without ever seeing those values directly.

Key Identity $$e([a]G_1, [b]G_2) = e(G_1, G_2)^{ab} = e([b]G_1, [a]G_2) = e([ab]G_1, G_2).$$ This "moves scalars around" freely between the arguments.

What does "in the exponent" mean?

Recall that $[a]G_1$ means $\underbrace{G_1 + G_1 + \cdots + G_1}_{a \text{ times}}$. The scalar $a$ is "locked inside" the curve point—given $[a]G_1$, you cannot extract $a$ (that's the discrete log problem). We say $a$ is "in the exponent" because the notation mirrors $g^a$ in multiplicative groups.

Now suppose someone gives you two curve points $A = [a]G_1$ and $B = [b]G_2$, and you want to check whether $a \cdot b = c$ for some known $c$—without knowing $a$ or $b$. Ordinary arithmetic can't help: you can't multiply curve points. But pairings can:

$$e(A, B) = e([a]G_1,\; [b]G_2) = e(G_1, G_2)^{ab}.$$

Meanwhile, you can independently compute $e(G_1, G_2)^c$. If $e(A, B) = e(G_1, G_2)^c$, then $ab = c$. You just verified a multiplication of hidden values using only their curve-point "encryptions."

Concrete Example — Verifying a Hidden Product

Suppose a prover claims they know $a, b$ such that $a \cdot b = 6$, and they give you $A = [a]G_1$, $B = [b]G_2$. You check:

$$e(A, B) \stackrel{?}{=} e([6]G_1,\; G_2).$$

By bilinearity, the left side equals $e(G_1,G_2)^{ab}$ and the right side equals $e(G_1,G_2)^6$. These are equal if and only if $ab = 6$. You verified the relationship without learning $a$ or $b$.

This is exactly how KZG verification works (Section 4): the verifier checks that $q(\tau) \cdot (\tau - \zeta) = f(\tau) - v$ (a polynomial equation in the secret $\tau$) by comparing pairings of curve points that "contain" these values in their exponents.

4. Polynomial Commitments (KZG)

A polynomial commitment scheme allows a prover to commit to a polynomial and later prove evaluations of it. This project uses the KZG (Kate-Zaverucha-Goldberg) scheme, which relies on pairings.

4.1 Trusted Setup (Structured Reference String)

A one-time setup generates the SRS. A secret $\tau$ (toxic waste, must be destroyed) is chosen, and the following are published:

$$\text{SRS} = \bigl(G_1,\; [\tau]G_1,\; [\tau^2]G_1,\; \dots,\; [\tau^d]G_1,\;\; G_2,\; [\tau]G_2\bigr)$$

where $G_1, G_2$ are generators of $\mathbb{G}_1, \mathbb{G}_2$ and $d$ is the maximum polynomial degree. Nobody knows $\tau$, only its "encrypted" forms on the curve.

4.2 Commit

To commit to a polynomial $f(X) = \sum_{i=0}^{d} c_i X^i$, the prover computes:

$$C = [f(\tau)]G_1 = \sum_{i=0}^{d} c_i \cdot [\tau^i]G_1.$$

The prover can compute this using only the SRS (without knowing $\tau$). The commitment $C$ is a single elliptic curve point—it is binding (the prover cannot change $f$ after committing) and hiding.

4.3 Open and Verify

Suppose the prover wants to convince the verifier that $f(\zeta) = v$ for a specific point $\zeta$ and value $v$. The protocol has three steps.

Step 1: The key algebraic idea

By the factor theorem, $f(\zeta) = v$ if and only if $(X - \zeta)$ divides $f(X) - v$ with no remainder. That is:

$$f(\zeta) = v \quad \iff \quad f(X) - v = (X - \zeta) \cdot q(X) \quad \text{for some polynomial } q(X).$$

We call $q(X)$ the quotient polynomial. It has degree $\deg(f) - 1$.

This is a polynomial identity, not just a point evaluation. The equation $f(X) - v = (X - \zeta) \cdot q(X)$ holds for every $X \in \mathbb{F}_p$, including the secret $X = \tau$. Substituting gives $f(\tau) - v = (\tau - \zeta) \cdot q(\tau)$, which is precisely what the verifier checks in step 3. The logical flow across the three steps is: $$\underbrace{f(\zeta) = v}_{\text{claim}} \;\iff\; \underbrace{f(X) - v = (X-\zeta)\cdot q(X) \;\forall X}_{\text{step 1: polynomial identity}} \;\Rightarrow\; \underbrace{f(\tau) - v = (\tau-\zeta)\cdot q(\tau)}_{\text{step 3: check at } X=\tau \text{ via pairing}}$$ The last arrow is $\Rightarrow$ rather than $\iff$ because a cheating prover could in principle fabricate values satisfying the equation at $X = \tau$ alone. However, $\tau$ is secret and fixed before the proof, so finding such values is computationally infeasible.
Why? (Factor Theorem) Dividing $f(X)$ by $(X - \zeta)$ gives $f(X) = (X - \zeta) \cdot q(X) + r$ where $r$ is a constant. Substituting $X = \zeta$: $$f(\zeta) = \underbrace{(\zeta - \zeta)}_{= \, 0} \cdot \, q(\zeta) + r = r.$$ So the remainder $r$ equals $f(\zeta)$. If $f(\zeta) = v$, then $r = v$, and $f(X) - v = (X-\zeta) \cdot q(X)$ exactly. If $f(\zeta) \neq v$, the remainder $r - v \neq 0$, and no such $q(X)$ exists.

Step 2: The prover's work

The prover computes $q(X)$ (which they can do since they know $f$), and sends its KZG commitment (same procedure as Section 4.2):

$$\pi = [q(\tau)]G_1.$$

Note: the prover uses the SRS to evaluate $q$ at the secret $\tau$ "in the exponent," just like the original commitment $C$.

Step 3: The verifier's pairing check

The verifier needs to check whether $q(\tau) \cdot (\tau - \zeta) = f(\tau) - v$ holds. All three quantities ($q(\tau)$, $\tau$, $f(\tau)$) are hidden in curve points, so the verifier uses a pairing. The check is:

$$e\bigl(\;\pi,\;\; [\tau]G_2 - [\zeta]G_2\;\bigr) \;\stackrel{?}{=}\; e\bigl(\;C - [v]G_1,\;\; G_2\;\bigr)$$

Let's unpack each side:

  • Left side: $\pi = [q(\tau)]G_1$ and $[\tau]G_2 - [\zeta]G_2 = [\tau - \zeta]G_2$, so by bilinearity: $$e([q(\tau)]G_1,\; [\tau - \zeta]G_2) = e(G_1, G_2)^{q(\tau)(\tau - \zeta)}.$$
  • Right side: $C - [v]G_1 = [f(\tau)]G_1 - [v]G_1 = [f(\tau) - v]G_1$, so: $$e([f(\tau) - v]G_1,\; G_2) = e(G_1, G_2)^{f(\tau) - v}.$$

The two sides are equal if and only if $q(\tau)(\tau - \zeta) = f(\tau) - v$, which is exactly the polynomial identity from Step 1. The verifier has confirmed $f(\zeta) = v$ without learning $f$, $\tau$, or $q$.

Theorem 4.1 — KZG Verification Correctness The pairing check succeeds if and only if $f(\zeta) = v$. The equivalence chain: $$\begin{aligned} &\underbrace{e(G_1, G_2)^{q(\tau)(\tau - \zeta)}}_{\text{left side}} \;=\; \underbrace{e(G_1, G_2)^{f(\tau) - v}}_{\text{right side}} \\[8pt] \iff\quad & q(\tau)\,(\tau - \zeta) \;=\; f(\tau) - v &\text{(compare exponents)}\\[4pt] \iff\quad & f(\zeta) = v &\text{(factor theorem)} \end{aligned}$$

How is the pairing $e(P, Q)$ actually computed?

Section 3 told us what pairings do (verify hidden-value equations via bilinearity), but not how. The construction may seem mysterious, so let's build intuition step by step.

The goal. We need a function $e(P, Q)$ that "multiplies exponents": $e([a]G_1, [b]G_2) = e(G_1, G_2)^{ab}$. How can we build such a thing from elliptic curve operations?

Key insight: eavesdrop on scalar multiplication. Recall from Section 2.3 that computing $[k]Q$ uses a double-and-add loop, which at each step draws a line on the curve (tangent for doubling, secant for addition—Section 2.2). Normally, we use these lines to find intersection points. But what if, instead, we evaluate each line at a separate point $P$ and multiply the results together? The resulting product would "encode" information about $k$ (the scalar) and $P$ simultaneously—exactly the bilinear structure we need.

That is Miller's algorithm. It runs the double-and-add loop on $Q \in \mathbb{G}_2$, but at every step, it also evaluates the line at $P \in \mathbb{G}_1$ and accumulates the result:

f = 1,  R = Q                      // accumulator and running point
for i = (num_bits - 2) down to 0:  // loop over bits of curve parameter
    f = f² · ℓR,R(P)                 // square f, multiply by tangent-at-R evaluated at P
    R = 2R                          // double R (same as point-addition doubling)
    if biti == 1:
        f = f · ℓR,Q(P)              // multiply by secant-through-R-and-Q evaluated at P
        R = R + Q                   // add Q
e(P, Q) = f(p¹²−1)/r                // final exponentiation

The line function $\ell_{A,B}(P)$ is the same tangent/secant line from the point addition in Section 2.2, but instead of finding where it hits the curve, we just plug in $P$'s coordinates and get a number:

$$\ell_{A,B}(P) = y_P - y_A - \lambda(x_P - x_A), \quad \lambda = \begin{cases} \dfrac{3x_A^2 + a}{2y_A} & A = B \text{ (tangent)}, \\[6pt] \dfrac{y_B - y_A}{x_B - x_A} & A \neq B \text{ (secant)}. \end{cases}$$

Since $P \in \mathbb{G}_1$ (over $\mathbb{F}_p$) and $A, B \in \mathbb{G}_2$ (over $\mathbb{F}_{p^2}$), the coordinates live in different fields, so the output is an element of $\mathbb{F}_{p^{12}}$ (the extension field from Section 1.2).

Final exponentiation. The loop produces $f \in \mathbb{F}_{p^{12}}$, but we need the result in the target group $\mathbb{G}_T$—the unique order-$r$ subgroup of $\mathbb{F}_{p^{12}}^*$ (where $r$ is the prime group order from Definition 2.2). Raising to the power $(p^{12} - 1)/r$ projects $f$ into this subgroup, the same way $a^{p-1} = 1$ in Fermat's little theorem (Section 1).

Why is the result bilinear? If we replace $P$ with $[a]P$, every line evaluation $\ell(\cdot)$ picks up a factor related to $a$. If we run the loop with $[b]Q$ instead of $Q$, the loop takes $b$ times as many effective steps. The multiplicative structure of the accumulator ensures these factors combine as $e(G_1, G_2)^{ab}$—exactly bilinearity. (The formal proof uses the theory of divisors on algebraic curves, but the intuition is: the accumulator records the "arithmetic trace" of a scalar multiplication, and scaling either input scales the trace proportionally.)

For BN254, the loop runs ~63 iterations (bits of $6u + 2$), and the entire KZG verification (two pairings) takes ~2ms.

5. Arithmetic Circuits & R1CS

Before we can prove a computation, we must express it as a system of algebraic constraints.

5.1 Arithmetic Circuits

An arithmetic circuit over $\mathbb{F}_p$ is a directed acyclic graph where:

  • Input nodes hold field elements (public inputs or private witnesses).
  • Internal nodes are either addition ($+$) or multiplication ($\times$) gates.
  • Output nodes produce the result.

5.2 From Circuits to Constraints (R1CS)

R1CS (Rank-1 Constraint System) is a way to encode an arithmetic circuit as a system of equations. Each multiplication gate becomes a constraint of the form:

$$(\vec{a} \cdot \vec{w}) \times (\vec{b} \cdot \vec{w}) = (\vec{c} \cdot \vec{w})$$

where $\vec{w}$ is the vector of all wire values (inputs, intermediates, outputs), and $\vec{a}, \vec{b}, \vec{c}$ are selector vectors. Addition is "free" and can be folded into the linear combinations.

Example — R1CS for $y = x^3 + x + 5$

Using the circuit from above, we introduce intermediate variables: $s_1 = x \cdot x = x^2$ and $s_2 = s_1 \cdot x = x^3$. The output is $y = s_2 + x + 5$. The witness vector is:

$$\vec{w} = (1,\; x,\; y,\; s_1,\; s_2)$$

where the leading 1 is a constant term. Each multiplication gate becomes one R1CS constraint:

Gate$\vec{a} \cdot \vec{w}$$\times$$\vec{b} \cdot \vec{w}$$=$$\vec{c} \cdot \vec{w}$
$s_1 = x \cdot x$$x$$\times$$x$$=$$s_1$

The third constraint uses a trick: addition is encoded as multiplication by 1, since $(s_2 + x + 5) \times 1 = y$. Written as selector vectors:

$$\vec{a}_3 = (5, 1, 0, 0, 1), \quad \vec{b}_3 = (1, 0, 0, 0, 0), \quad \vec{c}_3 = (0, 0, 1, 0, 0)$$

Verify: $\vec{a}_3 \cdot \vec{w} = 5 \cdot 1 + 1 \cdot x + 1 \cdot s_2 = s_2 + x + 5$, $\;\vec{b}_3 \cdot \vec{w} = 1$, $\;\vec{c}_3 \cdot \vec{w} = y$.

While R1CS is used in systems like Groth16, Halo2 uses a more expressive format called PLONKish arithmetization.

6. PLONKish Arithmetization

PLONKish is the constraint system used by Halo2. It generalizes R1CS by organizing computation into a table (like a spreadsheet) with custom gates and lookup arguments.

6.1 The Execution Trace Table

The computation is laid out in a table with $n$ rows and several types of columns:

Column TypeSymbolWho knows itRole
Advice$a_i$Prover onlyPrivate witness values; committed via KZG and opened at $\zeta$ in the proof
Instance$x_i$BothPublic inputs/outputs; verifier evaluates $x_i(\zeta)$ directly — no commitment needed
Fixed$f_i$BothArbitrary constants baked in at compile time (e.g. lookup table entries, constant weights); verifier evaluates $f_i(\zeta)$ directly
Selector$q_i$BothBinary ($0$ or $1$) fixed columns that activate/deactivate gates per row; a special case of fixed columns

All four column types can appear in gate constraint polynomials $g_j(X)$. Their evaluations at $\zeta$ come from different sources during verification:

  • $\{a_i(\zeta)\}$ — received from the prover inside $\pi$ (private, must be proven consistent with the commitments via SHPLONK).
  • $\{x_i(\zeta)\}$ — computed by the verifier directly from the public inputs (not in $\pi$). Used to anchor private computations to declared public values, e.g. $a(\omega^i) - x(\omega^i) = 0$ enforces that a private result equals the public output.
  • $\{f_i(\zeta)\}$, $\{q_i(\zeta)\}$ — computed by the verifier directly from the fixed circuit data (not in $\pi$). Fixed columns hold arbitrary constants; selectors are the binary special case that activate/deactivate individual gates.
Example — Arithmetic Gate A single PLONKish custom gate with advice columns $a, b, c$ and selectors $q_M, q_L, q_R, q_O, q_C$: $$q_M \cdot a \cdot b \;+\; q_L \cdot a \;+\; q_R \cdot b \;+\; q_O \cdot c \;+\; q_C = 0.$$ By setting different selector values, this one gate template can express:
  • Multiplication: $q_M=1, q_O=-1$, rest $0$ → $a \cdot b = c$.
  • Addition: $q_L=1, q_R=1, q_O=-1$ → $a + b = c$.
  • Constant: $q_L=1, q_C=-7$ → $a = 7$.

6.2 Custom Gates

PLONKish allows custom gates—polynomial constraints of arbitrary degree involving any combination of columns, even across adjacent rows (via "rotation"). For example, a gate can reference $a_i(\text{row}_j)$ and $a_i(\text{row}_{j+1})$ simultaneously.

In Halo2-base (used by this project), the primary gate is the FlexGate:

$$q \cdot \bigl(a \cdot b \cdot c_{\text{mul}} + a \cdot c_a + b \cdot c_b + c \cdot c_c + d \cdot c_d + \text{const}\bigr) = 0$$

This can encode any combination of multiply-and-add in a single gate.

6.3 Lagrange Interpolation

Before we can turn the execution trace into polynomials, we need a tool that converts a table of values into a polynomial. Lagrange interpolation does exactly this.

Definition 6.2 — Lagrange Interpolation Given $n$ distinct points $x_0, x_1, \dots, x_{n-1}$ and corresponding values $y_0, y_1, \dots, y_{n-1}$, there exists a unique polynomial $p(X)$ of degree $< n$ such that $p(x_i) = y_i$ for all $i$. It is: $$p(X) = \sum_{i=0}^{n-1} y_i \cdot L_i(X), \qquad \text{where} \quad L_i(X) = \prod_{\substack{j=0 \\ j \neq i}}^{n-1} \frac{X - x_j}{x_i - x_j}.$$ Each $L_i$ is a Lagrange basis polynomial: it equals 1 at $x_i$ and 0 at every other $x_j$.
Example — Interpolating 3 points

Find the polynomial through $(1, 3)$, $(2, 7)$, $(4, 15)$.

The Lagrange basis polynomials are:

$$\begin{aligned} L_0(X) &= \frac{(X-2)(X-4)}{(1-2)(1-4)} = \frac{(X-2)(X-4)}{3}, \\[6pt] L_1(X) &= \frac{(X-1)(X-4)}{(2-1)(2-4)} = \frac{(X-1)(X-4)}{-2}, \\[6pt] L_2(X) &= \frac{(X-1)(X-2)}{(4-1)(4-2)} = \frac{(X-1)(X-2)}{6}. \end{aligned}$$

So $p(X) = 3 \cdot L_0(X) + 7 \cdot L_1(X) + 15 \cdot L_2(X)$. Expanding and simplifying gives $p(X) = X^2 - X + 3$. Verify: $p(1) = 3$, $p(2) = 7$, $p(4) = 15$.

Why Lagrange basis? The trick is that $L_i(x_j) = 0$ when $j \neq i$ (because the numerator contains $(x_j - x_j) = 0$), and $L_i(x_i) = 1$ (because numerator equals denominator). So $p(x_i) = \sum_j y_j \cdot L_j(x_i) = y_i \cdot 1 + \sum_{j \neq i} y_j \cdot 0 = y_i$.

6.4 From Table to Polynomials

The execution trace table has $n$ rows. We associate each row $i$ with a root of unity $\omega^i$, where $H = \{1, \omega, \omega^2, \dots, \omega^{n-1}\}$ and $\omega$ is a primitive $n$-th root of unity in $\mathbb{F}_p$ (see Section 1.1).

Each column is then Lagrange-interpolated into a polynomial that "encodes" all its row values. For example, if the advice column has values $a_0, a_1, \dots, a_{n-1}$, the Lagrange formula from Section 6.3 gives us the unique polynomial $a(X)$ of degree $< n$ such that:

$$a(\omega^0) = a_0, \quad a(\omega^1) = a_1, \quad \dots, \quad a(\omega^{n-1}) = a_{n-1}.$$

The same is done for every column ($b(X)$, $c(X)$, $q_M(X)$, etc.). The special structure of the evaluation points (roots of unity) means this interpolation can be done efficiently using the Number Theoretic Transform (NTT)—the finite-field analogue of the FFT—in $O(n \log n)$ time instead of the naive $O(n^2)$.

Why $O(n \log n)$? The idea is analogous to fast exponentiation ($a^n$ in $O(\log n)$ by repeated squaring). Since $n = 2^k$, a polynomial can be split into its even- and odd-indexed coefficients: $a(X) = a_{\text{even}}(X^2) + X \cdot a_{\text{odd}}(X^2)$. The key property of roots of unity is that squaring the $n$-th roots gives the $(n/2)$-th roots, so each half is the same problem at half the size. This gives the recursion $T(n) = 2\,T(n/2) + O(n) = O(n \log n)$. It is also why Halo2 requires $n$ to be a power of two.

Now, a gate constraint like "the multiplication gate holds at row $i$" becomes $q_M(\omega^i) \cdot a(\omega^i) \cdot b(\omega^i) - c(\omega^i) = 0$. If this holds at every row, then the polynomial

$$g(X) = q_M(X) \cdot a(X) \cdot b(X) - c(X)$$

has every element of $H$ as a root: $g(\omega^i) = 0$ for all $i = 0, \dots, n-1$. Note that $X \in \mathbb{F}_p$ (not just $H$)—the constraint must hold on $H$, but $g(X)$ is a polynomial over all of $\mathbb{F}_p$, which is why the verifier can later check at a random $\zeta \notin H$.

By the factor theorem (Section 4.3), this means $(X - \omega^i)$ divides $g(X)$ for each $i$. Since all $n$ roots divide $g(X)$, the product

$$Z_H(X) = (X - 1)(X - \omega)(X - \omega^2) \cdots (X - \omega^{n-1}) = X^n - 1$$

divides $g(X)$. This polynomial $Z_H(X)$ is called the vanishing polynomial of $H$.

Why $Z_H(X) = X^n - 1$ The elements $1, \omega, \omega^2, \dots, \omega^{n-1}$ are precisely the $n$ solutions to $X^n = 1$ (by definition of $n$-th roots of unity: $(\omega^i)^n = (\omega^n)^i = 1^i = 1$). So $X^n - 1$ is a degree-$n$ monic polynomial that vanishes at all $n$ elements of $H$. But the product $(X-1)(X-\omega)\cdots(X-\omega^{n-1})$ is also a degree-$n$ monic polynomial that vanishes at exactly the same $n$ points. Since a degree-$n$ polynomial is uniquely determined by its $n$ roots and leading coefficient, the two must be equal.
Connection to KZG (Section 4.3): In KZG, we proved $f(\zeta) = v$ by showing $(X - \zeta)$ divides $f(X) - v$, i.e., one evaluation at one point. Here, we need to prove that a constraint holds at all $n$ rows simultaneously. The idea is the same, but instead of dividing by $(X - \zeta)$ (one root), we divide by $Z_H(X) = X^n - 1$ (all $n$ roots at once).

The prover demonstrates this by exhibiting a quotient polynomial $t(X)$ such that:

$$g(X) \;=\; q_M(X) \cdot a(X) \cdot b(X) - c(X) \;=\; t(X) \cdot Z_H(X).$$

The verifier then uses KZG to check that this identity holds—but not by checking every point. Instead, the verifier checks at a single random point $\zeta$. The Schwartz-Zippel lemma explains why this is sufficient.

6.5 The Schwartz-Zippel Lemma

A key question: if the prover gives us polynomials and claims they satisfy an identity $g(X) = t(X) \cdot Z_H(X)$, why is it enough to check this at a single random point $\zeta$? The answer is one of the most useful lemmas in theoretical computer science:

Theorem 6.1 — Schwartz-Zippel Lemma Let $h(X)$ be a nonzero polynomial of degree $d$ over a field $\mathbb{F}$. If $\zeta$ is chosen uniformly at random from a subset $S \subseteq \mathbb{F}$, then: $$\Pr[h(\zeta) = 0] \;\leq\; \frac{d}{|S|}.$$ A nonzero polynomial of degree $d$ has at most $d$ roots, so a random evaluation is unlikely to hit one.

How does this apply? Suppose the constraint polynomial identity does not hold: $g(X) - t(X) \cdot Z_H(X) \neq 0$. This difference polynomial has degree at most $D$ (a few times $n$). If the verifier picks $\zeta$ at random from $\mathbb{F}_p$, the probability that the check $g(\zeta) = t(\zeta) \cdot Z_H(\zeta)$ falsely passes is:

$$\Pr[\text{false accept}] \;\leq\; \frac{D}{p} \;\approx\; \frac{2^{20}}{2^{254}} \;\approx\; 2^{-234}.$$

This is astronomically small. In other words: if the identity is wrong anywhere, a random check catches it with overwhelming probability.

The power of random evaluation: Without Schwartz-Zippel, the verifier would need to check the polynomial identity at every point in $H$ (all $n$ rows)—defeating the purpose of succinctness. Schwartz-Zippel compresses $n$ checks into one random check, with negligible error probability. This is why the random challenge $\zeta$ in the Halo2 protocol (Section 7) is so important: it turns a polynomial identity over all of $H$ into a single-point verification.

6.6 Permutation Arguments (Copy Constraints)

Gate constraints check that each row satisfies a local equation. But circuits also need to ensure that a value at one position in the table equals a value at another position—for example, that the output of gate 1 is the same as an input to gate 5. A cell refers to a specific (column, row) entry in the execution trace table from Section 6.1. These equality requirements are called copy constraints, and they are enforced by a permutation argument.

The idea

Suppose we have a set of cells that should all contain the same value. In the execution trace, these cells might be at different rows and columns. We define a permutation $\sigma$ that maps each cell to the next cell in its "equality group"—forming a cycle. If cell $A$ should equal cells $B$ and $C$, then $\sigma(A) = B$, $\sigma(B) = C$, $\sigma(C) = A$.*

* In other words, $\sigma$ acts as a circular shift within each equality group. The full permutation is a collection of disjoint cycles: one cycle per group of cells that must share the same value. Cells with no copy constraint simply map to themselves ($\sigma(F) = F$, a trivial 1-cycle).

The claim "all cells in each cycle have equal values" is equivalent to: the column polynomials evaluated at the permuted positions give the same values as at the original positions.

Grand product argument

To check that two lists $(v_0, \dots, v_{n-1})$ and $(v_{\sigma(0)}, \dots, v_{\sigma(n-1)})$ are the same multiset (which, combined with the permutation structure, implies element-wise equality within each cycle), PLONK uses a grand product. Here $v_i = a(\omega^i)$ is the column polynomial evaluated at the $i$-th root of unity, i.e., the cell value at row $i$ (from Section 6.4). The verifier sends random challenges $\beta, \gamma$, and the prover constructs an accumulator polynomial $Z(X)$ such that:

$$Z(\omega^0) = 1, \qquad Z(\omega^{i+1}) = Z(\omega^i) \cdot \frac{v_i + \beta \cdot \omega^i + \gamma}{v_i + \beta \cdot \sigma(\omega^i) + \gamma}.$$

The key insight: if the values respect the permutation (i.e., $v_i = v_{\sigma(i)}$ within each cycle), then the product equals 1:

$$Z(\omega^n) = \prod_{i=0}^{n-1} \frac{v_i + \beta \cdot \omega^i + \gamma}{v_i + \beta \cdot \sigma(\omega^i) + \gamma} = 1.$$
Why the product is 1 Consider a single cycle $\sigma: \omega^a \to \omega^b \to \omega^c \to \omega^a$. Copy constraints force $v_a = v_b = v_c = v$. The factors from these three indices are: $$\begin{aligned} \frac{\text{Numerator}}{\text{Denominator}} &= \frac{(v + \beta\omega^a + \gamma)(v + \beta\omega^b + \gamma)(v + \beta\omega^c + \gamma)}{(v + \beta\,\sigma(\omega^a) + \gamma)(v + \beta\,\sigma(\omega^b) + \gamma)(v + \beta\,\sigma(\omega^c) + \gamma)} \\[6pt] &= \frac{(v + \beta\omega^a + \gamma)(v + \beta\omega^b + \gamma)(v + \beta\omega^c + \gamma)}{(v + \beta\omega^b + \gamma)(v + \beta\omega^c + \gamma)(v + \beta\omega^a + \gamma)} \\[6pt] &= 1. \end{aligned}$$ The denominator contains the same factors as the numerator, just reordered by $\sigma$. Since every index belongs to exactly one cycle, the full product is 1.

If any copy constraint is violated, the product will (with overwhelming probability over the random $\beta, \gamma$) not equal 1.

Example — Copy constraint

Suppose column $a$ at row 0 should equal column $b$ at row 2. In the permutation, the cell $(a, 0)$ is mapped to $(b, 2)$ and vice versa. If the prover puts $a(\omega^0) = 7$ and $b(\omega^2) = 7$, the corresponding factors in the grand product cancel, and $Z$ cycles back to 1. If the prover cheats and puts $b(\omega^2) = 9$, the factors don't cancel, and $Z(\omega^n) \neq 1$.

The polynomial $Z(X)$ is committed via KZG, and the recurrence relation and the boundary condition $Z(\omega^n) = 1$ are checked as additional polynomial constraints (folded into the quotient polynomial from Section 6.4).

Why randomness? Without $\beta$ and $\gamma$, a cheating prover could construct values that make the product equal 1 even when the permutation is violated (by finding cancellations in the product). The random challenges ensure that such cancellations happen with negligible probability (by Schwartz-Zippel from Section 6.5).

6.7 Lookup Arguments

Some constraints are hard to express as low-degree polynomial equations (e.g., "this value is a valid 8-bit integer"). Lookup arguments solve this: the prover shows that a set of values is contained in a predefined table.

Definition 6.1 — Lookup Argument Given a lookup table $T = \{t_1, t_2, \dots, t_N\}$ (a fixed column), a lookup argument proves that for each active row $i$, the advice value $a_i \in T$. In this project, the table is $T = \{0, 1, 2, \dots, 2^{\ell} - 1\}$ with $\ell = $ LOOKUP_BITS (typically 15). This efficiently proves range checks: $0 \leq x < 2^{\ell}$.

Why not just use polynomial gates?

Consider proving that a value $x$ is an 8-bit integer, i.e., $0 \leq x < 256$. Without lookups, you would need to decompose $x$ into 8 individual bits $b_0, \dots, b_7$, constrain each $b_i \in \{0,1\}$ (via $b_i(b_i - 1) = 0$), and verify $x = \sum_{i=0}^{7} b_i \cdot 2^i$. That is 8 extra witness variables and 9 constraints for a single range check. With a lookup table $T = \{0, 1, \dots, 255\}$, you prove $x \in T$ in a single step.

How lookups work (intuition)

The core idea is a set-membership test via sorted permutations. At a high level:

  1. Let $\vec{f} = (f_1, f_2, \dots, f_m)$ be the values the prover wants to look up, and $\vec{t} = (t_1, t_2, \dots, t_N)$ be the fixed table.
  2. The prover produces a sorted combination: merge $\vec{f}$ and $\vec{t}$ into a single sorted list $\vec{s}$. If every $f_i$ is indeed in $T$, then $\vec{s}$ is just $\vec{t}$ with some entries repeated.
  3. The verifier checks (via polynomial constraints) that:
    • $\vec{s}$ is a valid interleaving of $\vec{f}$ and $\vec{t}$ (a permutation argument).
    • $\vec{s}$ is sorted, and consecutive elements in $\vec{s}$ that differ must appear in $\vec{t}$ (no new values were introduced).

All of these checks are encoded as polynomial identities over the roots of unity, so they fit naturally into the PLONKish framework from Section 6.4.

The polynomial constraints

The prover constructs two auxiliary column polynomials: $A'(X)$ (the lookup values permuted into table order) and $S'(X)$ (the table values repeated to match). Three polynomial constraints enforce correctness:

1. Grand product (same idea as the permutation argument in Section 6.6). With random challenges $\beta, \gamma$, the prover builds an accumulator $Z(X)$:

$$Z(\omega \cdot X) \cdot (A'(X) + \beta) \cdot (S'(X) + \gamma) = Z(X) \cdot (f(X) + \beta) \cdot (t(X) + \gamma)$$

with boundary $Z(\omega^0) = 1$. This ensures $(A', S')$ is a valid permutation of $(f, t)$.

2. Boundary. The product must return to 1 after all rows:

$$L_0(X) \cdot (Z(X) - 1) = 0 \quad \text{on } H$$

where $L_0(X)$ is the Lagrange basis polynomial for row 0 (from Section 6.3: equals 1 at $\omega^0$, 0 elsewhere).

3. Sorting / adjacency. This is the key constraint:

$$(A'(X) - S'(X)) \cdot (A'(X) - A'(\omega^{-1} \cdot X)) = 0 \quad \text{on } H$$

At each row, either $A'(X) = S'(X)$ (the value matches the table at this position) or $A'(X) = A'(\omega^{-1} \cdot X)$ (the value equals the previous row—a repeated lookup). This ensures no value outside $T$ can appear in $A'$: every "run" of equal values must start at a position matching the table $S'$.

All three are polynomial identities that vanish on $H$, so they get folded into the quotient polynomial $t(X) \cdot Z_H(X)$ from Section 6.4.

Example — Range check via lookup

Suppose the table is $T = \{0, 1, 2, 3\}$ (2-bit range) and the prover wants to prove $f_1 = 2$ and $f_2 = 1$ are both in $T$. The sorted combination is $\vec{s} = (0, 1, 1, 2, 2, 3)$. Each consecutive pair in $\vec{s}$ either stays the same or increases by an amount that matches a gap in $T$, confirming no value outside $T$ was smuggled in.

If the prover tried $f_1 = 5$, the sorted list would contain $(\dots, 3, 5, \dots)$. The gap $3 \to 5$ skips over 4, which is not in $T$, and the polynomial constraint would fail.

Range checks are essential for fixed-point arithmetic (ensuring values don't overflow) and for comparison operations (is $a < b$?).

7. The Halo2 Proving System

Halo2 combines PLONKish arithmetization with KZG polynomial commitments to create a complete proof system. Here is the high-level protocol:

7.1 The Fiat-Shamir Transform

The polynomial commitment protocol from Section 4 is interactive: the verifier must send a random challenge $\zeta$, and the prover must respond. This requires the verifier to be online during proof generation. The Fiat-Shamir transform eliminates this interaction, producing a non-interactive proof.

Definition 7.1 — Fiat-Shamir Heuristic Instead of the verifier sampling a random challenge, the prover computes it as a hash of the transcript so far (all messages exchanged up to that point): $$\zeta = \text{Hash}(\text{transcript})$$ The transcript includes all commitments, public inputs, and any prior challenges. Since a cryptographic hash (Halo2 uses Blake2b or Poseidon) behaves like a random oracle, the prover cannot predict or manipulate $\zeta$ without changing the commitments, which would change the hash.

Here is how it works step by step:

  1. The prover commits to the witness polynomials and appends the commitments to a running transcript.
  2. When a random challenge is needed, the prover hashes the transcript to derive it: $\alpha = \text{Hash}(\text{transcript})$.
  3. The challenge $\alpha$ is appended to the transcript, and the process continues.
  4. The verifier independently reconstructs the transcript from the proof, re-derives all challenges via the same hashing, and checks everything matches.
Why is this secure? The random oracle model treats the hash function as a truly random function. A cheating prover would need to find commitments that, when hashed, produce a challenge that makes a false proof verify—but this requires inverting or finding collisions in the hash, which is computationally infeasible. The result: a proof that anyone can verify at any time, with no interaction.

7.2 Proof Generation (Prover)

The prover knows the private witness. It produces a proof consisting of the commitments and scalar evaluations listed below, which are appended to a public transcript as they are computed.

  1. Fill the table: The prover computes the witness and fills in all advice columns.
    Output: nothing sent yet; witness stays private.
  2. Commit to columns: Interpolate each advice column into a polynomial and compute KZG commitments: $$C_{a_i} = [a_i(\tau)]G_1.$$ Output: commitments $\{C_{a_i}\}$ — one curve point per advice column — appended to the transcript.
  3. Permutation accumulator: Derive random challenges $\beta, \gamma$ via Fiat-Shamir from the transcript so far (i.e. from $\{C_{a_i}\}$). Use them to compute the grand product polynomial $Z(X)$ from Section 6.6 that enforces copy constraints, and commit to it.
    Output: commitment $C_Z = [Z(\tau)]G_1$ appended to the transcript. ($\beta, \gamma$ are derived, not sent.)
  4. Construct constraint polynomial: Each $g_j(X)$ is one individual constraint polynomial—a gate constraint (e.g. $q_M(X)\cdot a(X)\cdot b(X) - c(X)$ for a multiplication gate), a lookup argument polynomial, or a permutation check polynomial—expressed as a polynomial over $\mathbb{F}_p$ that must vanish on every row in $H$. The random challenge $\alpha$ (derived via Fiat-Shamir from the transcript so far) is used to combine them into a single polynomial identity: $$\sum_j \alpha^j \cdot g_j(X) \equiv 0 \pmod{Z_H(X)}.$$ The $\alpha^j$ weighting ensures that satisfying the combined sum implies each individual $g_j(X)$ vanishes on $H$ with high probability (by the Schwartz–Zippel lemma).
    No new output yet; $\alpha$ is derived, not sent.
  5. Compute quotient polynomial: Compute $t(X) = \frac{\sum_j \alpha^j \cdot g_j(X)}{Z_H(X)}$. Because a degree-$D$ gate makes $g_j(X)$ have degree up to $D(n-1)$, the quotient $t(X)$ has degree up to $(D-1)(n-1)$, which exceeds the SRS maximum degree of $n-1$. To stay within the SRS, split $t(X)$ into $D-1$ chunks of degree $\leq n-1$ by grouping its coefficients: $$t(X) = t_0(X) + X^n \cdot t_1(X) + \cdots + X^{(D-2)n} \cdot t_{D-2}(X).$$ Commit to each chunk separately. At challenge point $\zeta$ the verifier reconstructs the full evaluation as $t(\zeta) = t_0(\zeta) + \zeta^n \cdot t_1(\zeta) + \cdots + \zeta^{(D-2)n} \cdot t_{D-2}(\zeta)$.
    Output: commitments $\{C_{t_i} = [t_i(\tau)]G_1\}$ appended to the transcript.
  6. Open evaluations: At a random challenge point $\zeta$ (derived via the Fiat-Shamir transform from Section 7.1), the prover evaluates all committed polynomials at $\zeta$ (and $\omega\zeta$ for polynomials that need a rotation), and provides a SHPLONK batched opening proof $\pi_{\mathrm{open}}$.
    Output: scalar evaluations $\{a_i(\zeta),\; Z(\zeta),\; Z(\omega\zeta),\; t_i(\zeta),\; \dots\}$ and the opening proof $\pi_{\mathrm{open}}$, all appended to the transcript.

The complete proof sent to the verifier is:

$$\pi \;=\; \Bigl(\;\underbrace{\{C_{a_i}\}}_{\text{step 2}},\; \underbrace{C_Z}_{\text{step 3}},\; \underbrace{\{C_{t_i}\}}_{\text{step 5}},\; \underbrace{\{a_i(\zeta)\},\; Z(\zeta),\; Z(\omega\zeta),\; \{t_i(\zeta)\},\; \pi_{\mathrm{open}}}_{\text{step 6}}\;\Bigr).$$

7.3 Verification (Verifier)

The verifier holds three kinds of data before checking the proof:

  • Proof $\pi$ (from the prover): commitments $\{C_{a_i}\}$, $C_Z$, $\{C_{t_i}\}$, scalar evaluations $\{a_i(\zeta)\}$, $Z(\zeta)$, $Z(\omega\zeta)$, $\{t_i(\zeta)\}$, and opening proof $\pi_{\mathrm{open}}$.
  • Public inputs — instance columns $\{x_i\}$ (known directly, not in $\pi$): the verifier interpolates each $x_i$ into a polynomial $x_i(X)$ and evaluates it at $\zeta$ itself. No commitment or opening proof is needed for instance columns.
  • Circuit data (fixed at compile time): selector polynomials $\{q_i(X)\}$ and fixed columns $\{f_i(X)\}$, also evaluated at $\zeta$ directly by the verifier.

The SRS is also public. The verifier holds no private witness.

  1. Reconstruct challenges: Re-run the Fiat-Shamir transcript in the same order as the prover (Section 7.1): hash $\{C_{a_i}\}$ to derive $\beta, \gamma$; hash $C_Z$ to derive $\alpha$; hash $\{C_{t_i}\}$ to derive $\zeta$. Any tampering with the commitments changes the derived challenges and causes the identity check in step 2 to fail.
  2. Check the constraint identity at $\zeta$: Verify: $$\sum_j \alpha^j \cdot g_j(\zeta) \stackrel{?}{=} t(\zeta) \cdot Z_H(\zeta).$$ The column evaluations are the inputs to each $g_j(\zeta)$, not $g_j(\zeta)$ itself — each constraint expression combines evaluations from multiple columns. For example, a multiplication gate constraint gives: $$g_j(\zeta) = \underbrace{q_M(\zeta)}_{\text{selector}} \cdot \underbrace{a(\zeta)}_{\text{advice}} \cdot \underbrace{b(\zeta)}_{\text{advice}} - \underbrace{c(\zeta)}_{\text{advice}}$$ The same column evaluation (e.g. $q_M(\zeta)$) can appear in multiple $g_j$'s, and one $g_j$ spans columns of different types. The verifier assembles all the inputs from three sources:
    • received scalars $\{a_i(\zeta)\}$, $Z(\zeta)$, $Z(\omega\zeta)$ — from $\pi$ (private witness evaluations);
    • $\{x_i(\zeta)\}$ — computed directly from the public instance columns (not in $\pi$);
    • $\{q_i(\zeta)\}$, $\{f_i(\zeta)\}$ — computed directly from the fixed circuit data.
    It also computes $Z_H(\zeta) = \zeta^n - 1$ directly and reconstructs $t(\zeta)$ from the received chunk evaluations: $$t(\zeta) = t_0(\zeta) + \zeta^n \cdot t_1(\zeta) + \cdots + \zeta^{(D-2)n} \cdot t_{D-2}(\zeta).$$ This step is purely finite field arithmetic in $\mathbb{F}_p$ — no elliptic curve points appear. The commitments $\{C_{a_i}\}$, $C_Z$, $\{C_{t_i}\}$ are not used here; they are only needed in step 3.
  3. Verify opening proofs: This step extends the basic KZG pairing check from Section 4.3 to all polynomials at once using SHPLONK.

    Let $\{p_i\}$ denote all committed polynomials ($\{a_i\}$, $Z$, $\{t_i\}$) with commitments $\{C_i\}$. Each $p_i$ is opened at a set $\mathcal{O}_i \subseteq \{\zeta,\, \omega\zeta\}$ with claimed evaluations forming a remainder polynomial $r_i(X)$ — a constant if $|\mathcal{O}_i|=1$, or a line through the two claimed values if $|\mathcal{O}_i|=2$.

    1. Derive batching challenge $y$ via Fiat-Shamir from the transcript.
    2. Compute the batched commitment using $y$ and the claimed scalar evaluations: $$F = \sum_i y^i \cdot \bigl(C_i - [r_i(\tau)]G_1\bigr),$$ where $[r_i(\tau)]G_1$ is computed from the SRS using the claimed scalars (no knowledge of $\tau$ needed).
    3. Single pairing check: Check: $$e\!\left(\pi_{\mathrm{open}},\;\; [(\tau-\zeta)(\tau-\omega\zeta)]G_2\right) \stackrel{?}{=} e\!\left(F,\;\; G_2\right).$$ The two sides are equal if and only if every claimed evaluation is genuine.

    Without this check, a cheating prover could send false scalars that satisfy the identity in step 2 but do not correspond to any valid witness.

This project uses the SHPLONK multi-opening strategy, which batches all polynomial openings into a single pairing check for efficiency.

8. Fixed-Point Arithmetic in $\mathbb{F}_p$

Finite fields have no fractions—there is no $0.5$ in $\mathbb{F}_p$. To do machine learning (which needs decimal numbers), this project uses fixed-point quantization: represent real numbers as scaled integers.

8.1 Quantization Scheme

Definition 8.1 — Fixed-Point Representation Choose a precision parameter $P$ (number of fractional bits). A real number $x \in \mathbb{R}$ is represented as: $$x_q = \lfloor x \cdot 2^P \rceil \in \mathbb{F}_p$$ where $\lfloor \cdot \rceil$ denotes rounding to the nearest integer. To recover the approximate real value: $$\hat{x} = x_q / 2^P.$$ The scaling factor $S = 2^P$ is called the quantization scale.

This project supports $P \in [32, 63]$. With $P = 32$, we get a "32.32" format: 32 bits for the integer part and 32 bits for the fractional part, giving roughly 9 decimal digits of precision. The upper bound $P \leq 63$ is imposed by the BN254 field size: intermediate products $x_q \cdot y_q$ reach $2^{2P}$, which must stay below $p \approx 2^{254}$, so $2P \ll 254$, giving $P \leq 63$ with margin.

Example ($P = 32$) $\pi \approx 3.14159265$: $$\pi_q = \lfloor 3.14159265 \times 2^{32} \rceil = \lfloor 3.14159265 \times 4294967296 \rceil = 13{,}493{,}037{,}705.$$ De-quantization: $13{,}493{,}037{,}705 / 4{,}294{,}967{,}296 \approx 3.14159265$.
Quantization error. Rounding $x \cdot 2^P$ to the nearest integer introduces an error of at most $\frac{1}{2}$ in the last bit: $$|x - \hat{x}| = \frac{|x_q - x \cdot 2^P|}{2^P} \leq \frac{1}{2^{P+1}}.$$ With $P = 32$ this is $\approx 2.3 \times 10^{-10}$; with $P = 63$ it is $\approx 5.4 \times 10^{-20}$. The choice of $P$ trades range against precision: larger $P$ gives finer resolution but shrinks the safe integer range from $(-2^{2P}, 2^{2P})$ toward the field boundary.

8.2 Handling Negative Numbers

Since $\mathbb{F}_p$ contains only elements $\{0, 1, \dots, p-1\}$, negative numbers use modular wraparound — exactly like two's complement, but modulo $p$ instead of $2^{64}$:

$$\text{Encode}(-x) = p - x_q \equiv -x_q \pmod{p}.$$
Example — Encoding $-\pi$ ($P = 32$) $\pi_q = 13{,}493{,}037{,}705$. The BN254 prime is $p = 21888242871839275222246405745257275088548364400416034343698204186575808495617$. So: $$(-\pi)_q = p - 13{,}493{,}037{,}705 = 21888242871839275222246405745257275088548364400416034343684711148870123457912.$$ This large number, when subtracted from $p$ in the field, recovers $\pi_q$.

The chip defines a negative threshold:

$$\text{negative\_point} = p - 2^{2P+1} + 1.$$

Any value $v > \text{negative\_point}$ is interpreted as negative, and its real value is $-(p - v)/2^P$. The valid range is $(-2^{2P},\; 2^{2P})$.

Sign check in circuit. To determine the sign of a value $v$ without branching, the chip uses a range-chip comparison against the fixed constant negative_point: it decomposes $v - \text{negative\_point}$ into bits via lookup arguments (the same mechanism as is_less_than in §9.4.3) and reads off whether the result is positive or negative. This produces a binary sign bit $s \in \{0,1\}$ usable in subsequent gate constraints.

8.3 Arithmetic Operations

Addition and Subtraction

These work directly in $\mathbb{F}_p$ because quantization is linear:

$$x_q + y_q = (x + y)_q, \qquad x_q - y_q = (x - y)_q.$$

No rescaling is needed. The only risk is overflow: if $|x| + |y| \geq 2^{2P}$ the result wraps around the field and is misinterpreted. The valid range $(-2^{2P}, 2^{2P})$ must be respected by the caller.

Multiplication

Multiplying two quantized values gives a double-scaled result:

$$x_q \cdot y_q = (x \cdot 2^P)(y \cdot 2^P) = xy \cdot 2^{2P}.$$

We must rescale by dividing by $2^P$ and discarding the remainder $r$:

$$x_q \cdot y_q = (xy)_q \cdot 2^P + r, \qquad 0 \leq r < 2^P.$$ $$\text{qmul}(x_q, y_q) = (xy)_q = \frac{x_q \cdot y_q - r}{2^P}.$$

Circuit structure for qmul:

  1. Compute the raw product $N = |x_q| \cdot |y_q|$ using a single FlexGate multiply cell.
  2. Call div_mod(N, 2^P): the range chip finds quotient $q$ and remainder $r$ such that $N = q \cdot 2^P + r$, and enforces $0 \leq r < 2^P$ by decomposing $r$ into $P/15$ lookup chunks of 15 bits each.
  3. Determine the output sign: $\text{sign}(xy) = \text{sign}(x) \oplus \text{sign}(y)$. Extract each sign bit via the §8.2 sign-check gadget, XOR them (one gate), and conditionally negate $q$ if the result is negative.
  4. Enforce consistency: a gate constraint checks $|x_q| \cdot |y_q| = q \cdot 2^P + r$, binding the witness.

The truncation of $r$ introduces a rounding error of at most $1/2^P$ per multiplication (see §8.4).

Division

Division requires pre-scaling the numerator to maintain precision after the integer divide:

$$\text{qdiv}(x_q, y_q) = \frac{x_q \cdot 2^P}{y_q} = \frac{x \cdot 2^{2P}}{y \cdot 2^P} = \frac{x}{y} \cdot 2^P = (x/y)_q.$$

Circuit structure for qdiv:

  1. Pre-scale: compute $N = |x_q| \cdot 2^P$ (a bit-shift, implemented as a multiply-by-constant gate — 1 row).
  2. Call div_mod_var(N, |y_q|): finds quotient $q$ and remainder $r$ with $N = q \cdot |y_q| + r$ and $0 \leq r < |y_q|$. The remainder bound is enforced by a variable-modulus range check (more expensive than fixed-modulus, ~15 rows).
  3. Restore sign: $\text{sign}(x/y) = \text{sign}(x) \oplus \text{sign}(y)$, same XOR-and-negate gadget as qmul.
  4. Consistency gate: $|x_q| \cdot 2^P = q \cdot |y_q| + r$.

8.4 Precision and Error Analysis

Each arithmetic operation introduces a small truncation error from discarding the remainder. These errors accumulate as operations are chained.

Per-operation error

Let $\tilde{z}$ denote the computed fixed-point result and $z$ the exact real value. For each operation:

  • Addition/subtraction: no truncation, so $|\tilde{z} - z| = 0$ (beyond the input quantization error).
  • Multiplication: $|\widetilde{xy} - xy| \leq 2^{-P}$, since the dropped remainder satisfies $0 \leq r < 2^P$, i.e., error $r / 2^{2P} < 2^{-P}$ in real units.
  • Division: $|\widetilde{x/y} - x/y| \leq |y|^{-1} \cdot 2^{-P}$. Division by a small $|y|$ amplifies the error.

Error accumulation

For a chain of $k$ multiplications the errors add (they do not multiply, since each introduces an independent additive error):

$$\left|\tilde{z} - z\right| \leq k \cdot 2^{-P} + \varepsilon_{\text{input}}$$ where $\varepsilon_{\text{input}} \leq 2^{-(P+1)}$ is the initial quantization error. For $k = 100$ multiplications at $P = 32$: $$\text{accumulated error} \leq 100 \times 2^{-32} \approx 2.3 \times 10^{-8}.$$

This is well within acceptable tolerance for ML inference. For training (many more multiplications and divisions), larger $P$ (e.g. $P = 63$) is recommended to prevent gradient divergence.

Why not floating point? IEEE 754 floating point uses a variable exponent to maintain relative precision across a wide dynamic range. Fixed-point uses a constant exponent $2^P$, so precision is absolute rather than relative — values near zero have the same absolute error as large values. This is a limitation for ML models with large weight variance, but the simplicity of fixed-point (only integer arithmetic, no exponent tracking) makes it far cheaper to express as ZK circuit constraints.

9. Polynomial Function Approximation

Machine learning needs transcendental functions like $e^x$, $\log x$, $\sin x$, and $\sigma(x) = 1/(1+e^{-x})$. These cannot be computed exactly in a finite field—but they can be approximated with polynomials, which are native to our constraint system.

9.1 The Remez Algorithm

The Remez exchange algorithm finds the polynomial of degree $d$ that minimizes the maximum approximation error (the $L^\infty$ or minimax error) over an interval $[a, b]$:

$$p^*(x) = \arg\min_{p \in \mathcal{P}_d} \;\max_{x \in [a,b]}\; |f(x) - p(x)|.$$
Theorem 9.1 — Chebyshev Equioscillation The best minimax polynomial $p^*$ of degree $d$ is characterized by the fact that the error $f(x) - p^*(x)$ equioscillates: it attains its maximum absolute value at least $d+2$ times with alternating signs. This is a consequence of the Chebyshev equioscillation theorem.

This project uses Remez-optimized polynomials (generated via reference/remez.py) with the following specifications:

9.2 Horner's Method

Evaluating a polynomial $p(x) = c_0 + c_1 x + c_2 x^2 + \cdots + c_d x^d$ naively requires $O(d^2)$ multiplications. Horner's method reduces this to $d$ multiplications:

$$p(x) = c_0 + x\bigl(c_1 + x\bigl(c_2 + \cdots + x(c_{d-1} + x \cdot c_d)\cdots\bigr)\bigr).$$

In circuit terms, this is a chain of $d$ multiply-add operations, each costing one qmul and one qadd. The implementation in FixedPointChip::polynomial follows exactly this pattern.

9.3 Range Reduction

Polynomial approximations only work well on bounded intervals. To handle arbitrary inputs, we use range reduction:

Exponential: $2^x$

Decompose $x = n + f$ where $n = \lfloor x \rfloor$ (integer part) and $f \in [0, 1)$ (fractional part). Then:

$$2^x = 2^n \cdot 2^f.$$
  • $2^n$ is computed by table lookup (selecting from a precomputed array of powers of two).
  • $2^f$ is computed by the degree-12 Remez polynomial.

For negative $x$: $2^{-|x|} = 1 / 2^{|x|}$, computed via qdiv.

Natural exponential uses: $e^x = 2^{x / \ln 2}$.

Logarithm: $\log_2 x$

Find $n$ such that $2^n \leq x < 2^{n+1}$. Then normalize $x' = x / 2^{n-1} \in [2, 4)$ and:

$$\log_2 x = \log_2 x' + (n - 1).$$

The polynomial approximates $\log_2$ on $[2, 4)$, and the shift $(n-1)$ is added back. Natural logarithm uses: $\ln x = \log_2 x / \log_2 e$.

Trigonometric: $\sin x$

Reduce $x$ to $[0, 2\pi)$ via modular reduction: $x' = |x| \bmod 2\pi$. Then:

  • If $x' \in [0, \pi)$: compute $\sin(x')$ directly via the degree-14 polynomial.
  • If $x' \in [\pi, 2\pi)$: use $\sin(x') = -\sin(x' - \pi)$.

Finally, $\sin(-x) = -\sin(x)$ restores the sign. Cosine and tangent are derived:

$$\cos x = \sin(x + \pi/2), \qquad \tan x = \sin x / \cos x.$$

9.4 Circuit Gadgets for ML Primitives

Sections 8 and 9.1–9.3 provide the arithmetic building blocks (qmul, qdiv, Horner chains, range reduction). This section assembles them into reusable gadgets that the ML circuits in Section 10 call directly.

9.4.1 inner_product — Dot Product

Computes $\mathbf{u} \cdot \mathbf{v} = \sum_{i=1}^d u_i v_i$ for two quantized vectors of length $d$.

Circuit structure:

  1. For each $i = 1, \dots, d$: one qmul gate computes $p_i = u_i \cdot v_i$ (~10 rows each).
  2. A running accumulator adds products: $A_1 = p_1$, $A_i = A_{i-1} + p_i$ using $d-1$ qadd gates (1 row each).
  3. Copy constraints wire each $p_i$ output cell to the corresponding accumulator input cell, enforcing no intermediate value is swapped.

Row count: $d \times 10 + (d-1) \approx 11d$ rows. Output: $A_d = (\mathbf{u}\cdot\mathbf{v})_q$.

9.4.2 sigmoid — Logistic Activation

Computes $\sigma(z) = \frac{1}{1+e^{-z}}$ for a quantized scalar $z$.

Circuit structure (chained via copy constraints):

  1. qneg: $z' = -z$ — one subtraction gate (1 row).
  2. qexp: $e' = e^{z'} = 2^{z'/\ln 2}$ — range reduction splits $z' = n + f$, table lookup gives $2^n$, degree-12 Horner gives $2^f$, one qmul combines them (~200 rows).
  3. qadd: $d' = 1 + e'$ — one addition gate (1 row). The constant $1_q = 2^P$ is stored in a fixed column.
  4. qdiv: $\hat{y} = 1_q / d'$ — pre-scale, div_mod_var, sign handling (~20 rows).

Row count: ~222 rows. Output: $\hat{y}_q = \sigma(z)_q \in [0, 1]$.

9.4.3 is_less_than — Integer Comparison

Computes the boolean $[a < b]$ for two quantized values $a, b$, returning $1_q$ if true and $0$ otherwise.

Circuit structure:

  1. Compute $\Delta = b - a$ using qsub (1 row). If $b > a$ then $\Delta > 0$; if $b \leq a$ then $\Delta \leq 0$ (wraps around modulo $p$).
  2. Determine the sign of $\Delta$ by checking whether $\Delta > \text{negative\_point}$ (Section 8.2). This decomposes $\Delta$ into its constituent bits via the range chip's lookup table $\{0,\dots,2^{15}-1\}$, consuming ~13 rows.
  3. Output a binary cell $s$: if $\Delta$ is in the positive range, $s = 1$; otherwise $s = 0$. Enforced by a gate constraint $s \cdot (s-1) = 0$.

Row count: ~15 rows. Output: $s \in \{0, 1\}$.

9.4.4 select — Arithmetic Multiplexer

Given a binary selector $s \in \{0,1\}$ and two values $A$, $B$, computes $\text{out} = s \cdot A + (1-s) \cdot B$.

Circuit structure:

  1. qmul: $t_1 = s \cdot A$ (~10 rows). Since $s \in \{0,1\}$, no rescaling is needed; a specialized gate skips the $\div 2^P$ step (1 row).
  2. qsub: $\bar{s} = 1 - s$ (1 row).
  3. qmul: $t_2 = \bar{s} \cdot B$ (1 row, same specialized gate).
  4. qadd: $\text{out} = t_1 + t_2$ (1 row).

Row count: ~4 rows. Output: $A$ if $s=1$, else $B$. Used in decision tree branching to select the next node index without revealing which branch was taken.

9.5 Complete Function Reference

This section walks through every function in FixedPointChip, explaining the algorithm, circuit structure, and cost. Notation: $S = 2^P$ is the quantization scale, subscript $q$ denotes a quantized field element, and all operations below live in $\mathbb{F}_p$ with $p$ the BN254 prime.

9.5.1 Basic Arithmetic

qadd(a, b) — $\hat{a} + \hat{b}$

Delegates directly to the PLONKish addition gate: $\text{out} = a + b \pmod{p}$. No rescaling needed because both operands share the same scale $S$: $a_q + b_q = (a+b) \cdot S$. Cost: 1 row.

qsub(a, b) — $\hat{a} - \hat{b}$

Delegates to the subtraction gate: $\text{out} = a - b \pmod{p}$. Negative results wrap around modulo $p$ and are interpreted via the negative-point convention (§8.2). Cost: 1 row.

qmul(a, b) — $\hat{a} \cdot \hat{b}$

Multiplying two scaled values produces $a_q \cdot b_q = (ab) \cdot S^2$, which is one factor of $S$ too large. The implementation:

  1. Compute raw product $P = a \cdot b$ via a multiplication gate.
  2. Call signed_div_scale (see §9.5.2) to compute $\lfloor P / S \rfloor$ with correct sign handling, yielding $\text{out}_q = (ab) \cdot S$.

Cost: ~12 rows (1 mul + ~11 for signed_div_scale).

qdiv(a, b) — $\hat{a} / \hat{b}$

Division must restore the scale: $(a_q / b_q) \cdot S = (a/b) \cdot S$. Steps:

  1. Extract signs via is_neg; compute absolute values $|a|$, $|b|$.
  2. Pre-multiply: $a' = |a| \cdot S$ (this makes $a' / |b|$ land at the right scale).
  3. Compute the unsigned quotient $q = \lfloor a' / |b| \rfloor$ via div_mod_var (range chip).
  4. Restore sign: $\text{out} = q$ if $\text{sign}(a) = \text{sign}(b)$, else $-q$, using bit_xor + cond_neg.

Cost: ~20 rows.

qmod(a, b) — $\hat{a} \bmod \hat{b}$

Computes the fixed-point remainder. Constraint: $b$ must be positive. Steps:

  1. Assert $b > 0$ (check that is_neg(b) $= 0$).
  2. Compute $|a|$ and call div_mod_var to get remainder $r = |a| \bmod b$.
  3. If $a < 0$: return $b - r$ (so the result is in $[0, b)$ regardless of sign). Uses a select gate on the sign bit.

Cost: ~15 rows.

qsum(a[]) — $\sum_i \hat{a}_i$

Delegates to the gate's sum helper, which chains addition gates over the input array. Cost: $n-1$ rows for $n$ inputs.

inner_product(a[], b[]) — $\mathbf{a} \cdot \mathbf{b}$

For vectors of length $d$: calls qmul(a[i], b[i]) for each $i$, then accumulates with qadd. Cost: $\approx 11d$ rows (§9.4.1).

9.5.2 Sign, Absolute Value, and Clipping

is_neg(a) — returns $1$ if $a < 0$, else $0$

In BN254 fixed-point, a negative number satisfies $a > \text{negative\_point} = p - 2^{2P+1}$. The circuit:

  1. Right-shift $a$ by $2P+1$ bits using div_mod with divisor $2^{2P+1}$, yielding the high bits $h$.
  2. If $h = 0$, the number is non-negative; if $h \neq 0$, it is negative.
  3. Return $\texttt{not}(\texttt{is\_zero}(h))$.

Cost: ~13 rows (range decomposition).

sign(a) — returns $+1_q$ or $-1_q$

Calls is_neg(a) to get a binary flag, then uses a select gate to output the quantized constant $1_q = S$ or $-1_q = p - S$. Cost: ~15 rows.

cond_neg(a, is_neg) — returns $-a$ if is_neg $= 1$, else $a$

Computes $-a \pmod p$ unconditionally, then selects between $-a$ and $a$ using the select gate on the precomputed is_neg bit. Cost: 2 rows.

qabs(a) — $|\hat{a}|$

Computes is_neg(a), negates $a$ unconditionally to get $a'$, then selects: $\text{out} = a'$ if negative, else $a$. Cost: ~15 rows.

clip(a) — overflow clamping to $(-2^{2P}, 2^{2P})$

After operations that may overflow the valid range (e.g., intermediate sums), clip reduces the result modulo $2^{2P}$:

  1. Extract sign, compute $|a|$.
  2. Compute $|a| \bmod 2^{2P}$ via div_mod.
  3. Restore sign via cond_neg.

Warning: assumes $|a| < 2^{2P+1}$; silent overflow if the input is too large. Cost: ~20 rows.

bit_xor(a, b) — bitwise XOR for single bits

Asserts $a, b \in \{0,1\}$, computes $a + b$, and checks equality with $1$ via is_equal. The identity $a \oplus b = [a + b = 1]$ holds for bits. Cost: ~5 rows.

qmax(a, b) — $\max(\hat{a}, \hat{b})$

Computes $\Delta = a - b$; calls is_neg on $\Delta$ to get a bit $s$. Uses select: return $b$ if $s=1$ (meaning $a < b$), else $a$. Cost: ~18 rows.

qmin(a, b) — $\min(\hat{a}, \hat{b})$

Same as qmax but swaps the two outputs of select: return $a$ if $s=1$, else $b$. Cost: ~18 rows.

signed_div_scale(a) (internal helper)

Computes $(q, r)$ such that $a = q \cdot S + r$, $r \in [0, S)$, handling both positive and negative $a$ correctly in $\mathbb{F}_p$:

  1. Detect sign by comparing $\text{fe\_to\_biguint}(a)$ against $2^{252}$ (off-circuit hint).
  2. For negative $a$: compute $a_{\text{abs}} = p - a + 1$, then ceiling-divide: $q = \lceil a_{\text{abs}} / S \rceil$ taken modulo $p$ to represent $-\lceil a_{\text{abs}} / S \rceil$.
  3. For positive $a$: standard floor division.
  4. Assign the decomposition $[r, S, q, a]$ into a constraint region enforcing $r + S \cdot q = a$ via a single multiplication gate row.
  5. Range-check $r \in [0, S)$ and $|q| < 2^{3P}$.

Cost: ~11 rows.

check_power_of_two(pow2_exponent, exponent) (internal helper)

Proves in-circuit that pow2_exponent $= 2^{\texttt{exponent}}$:

  1. Decompose pow2_exponent into its $2P$ bits.
  2. Assert the sum of bits equals 1 (exactly one bit set).
  3. Use select_from_idx to extract the bit at position exponent and assert it is 1.

Cost: ~$2P$ rows.

9.5.3 Exponential and Logarithm

qexp2(a) — $2^{\hat{a}}$

Uses the range reduction $2^a = 2^n \cdot 2^f$ where $n = \lfloor a \rfloor$ and $f \in [0, 1)$.

  1. Split: div_mod(|a|, S) gives integer part $n$ (= int_part) and fractional part $f \cdot S$ (= frac_part).
  2. Table lookup: select $2^n$ from the precomputed pow_of_two array (length $254$) using select_from_idx(n). This witnesses the value without constraint; check_power_of_two is not called here since the table is fixed at setup time.
  3. Polynomial: evaluate the degree-12 Remez polynomial for $2^f$ on frac_part (§9.1).
  4. Combine: $\text{res\_pos} = 2^n \cdot 2^f$ via mul. Note: both $2^n$ and $2^f_q$ are fixed-point, so their raw product is at scale $S^2$; however $2^n$ here is the raw integer (not quantized), so the product is already at scale $S$.
  5. Negative input: if $a < 0$, output $1 / \text{res\_pos}$ via qdiv; otherwise output res_pos. Selected by is_neg(a).

Cost: ~240 rows (split ~13, table select $O(2P)$, Horner 12×qmul+qadd, qdiv ~20).

qexp(a) — $e^{\hat{a}}$

Uses the identity $e^x = 2^{x / \ln 2}$:

  1. Load constant $\ln 2 \approx 0.6931\ldots$ (quantized).
  2. Compute $x' = x / \ln 2$ via qdiv.
  3. Return qexp2(x').

Cost: ~260 rows.

qlog2(a) — $\log_2 \hat{a}$   (inspired by libm log.rs)

Mirrors the classical libm approach: find the integer part of the exponent by locating the most-significant bit, normalize to a canonical interval, then apply a polynomial approximation.

  1. Domain check: assert $a > 0$ — check is_neg(a) $= 0$ and is_zero(a) $= 0$; combine with or and assert constant $0$.
  2. MSB detection (off-circuit hint): scan the little-endian bit representation of $a$'s value to find the index of the highest set bit. Call this $\texttt{num\_digits}$. Then $2^{\texttt{num\_digits}} \leq a < 2^{\texttt{num\_digits}+1}$, so set $n = \texttt{num\_digits}$ and $\texttt{exp2} = n + 1$.
  3. In-circuit verification of the hint:
    1. Witness $\texttt{pow1} = 2^n$ and $\texttt{pow2} = 2^{n+1}$; load as Witness cells.
    2. Call check_power_of_two(pow1, n) and check_power_of_two(pow2, n+1) to prove the witnesses are correct powers of two.
    3. Use range comparisons to assert $\texttt{pow1} \leq a < \texttt{pow2}$ (two is_less_than + is_equal + and/or gates).
  4. Normalize to $[2, 4)$ in fixed-point: compute $\texttt{shift} = (P+2) - \texttt{exp2}$. Since $a \in [2^n, 2^{n+1})$, shifting by shift maps $a$ into $[2^{P+1}, 2^{P+2})$, which represents the interval $[2, 4)$ at scale $S = 2^P$.
    • If $\texttt{shift} \geq 0$: $a_{\text{norm}} = a \cdot 2^{\texttt{shift}}$ (left shift = multiply).
    • If $\texttt{shift} < 0$: $a_{\text{norm}} = a / 2^{|\texttt{shift}|}$ (right shift = div_mod_var).
    Both branches are witnessed; is_neg(shift) selects the correct one.
  5. Polynomial evaluation: apply the degree-14 Remez polynomial (§9.1, interval $[2, 4)$) to $a_{\text{norm}}$, giving $\log_2(a_{\text{norm}}) \in [1, 2)$ at scale $S$.
  6. Integer correction: $\log_2 a = \log_2 a_{\text{norm}} - \texttt{shift}$. Convert $-\texttt{shift}$ to fixed-point by multiplying by $S$: $\texttt{log\_shift\_q} = (-\texttt{shift}) \cdot S$. Final result: $\texttt{log\_a\_norm} + \texttt{log\_shift\_q}$.

Cost: ~$2P + 290$ rows (domain check ~3, MSB verify ~$4P$, shift ~10, Horner 14×, integer add ~2).

qlog(a) — $\ln \hat{a}$

Uses $\ln x = \log_2 x / \log_2 e$:

  1. Load constant $\log_2 e \approx 1.4427\ldots$ (quantized).
  2. Compute qlog2(a).
  3. Return qdiv(log2_a, log2e).

Cost: qlog2 cost + ~20 rows.

9.5.4 Trigonometric Functions

qsin(a) — $\sin \hat{a}$

Uses periodic range reduction and the identity $\sin(-x) = -\sin(x)$:

  1. Extract sign: $s = \texttt{is\_neg}(a)$; compute $|a|$ via qabs.
  2. Reduce to $[0, 2\pi)$: $a' = |a| \bmod 2\pi$ via qmod.
  3. Split at $\pi$: compute $a'' = a' - \pi$ via qsub and check $\texttt{is\_neg}(a'')$.
    • If $a'' < 0$ (i.e., $a' \in [0, \pi)$): evaluate the degree-14 polynomial directly on $a'$.
    • If $a'' \geq 0$ (i.e., $a' \in [\pi, 2\pi)$): evaluate the polynomial on $a''$, then negate (using $\sin(x + \pi) = -\sin(x)$).
    Both branches are computed; select picks the correct one.
  4. Restore sign: $\sin(a) = -\sin(|a|)$ if $a < 0$, via cond_neg(result, s).

Cost: ~460 rows (2× Horner degree-14 + qmod + 2× qabs + selects).

qcos(a) — $\cos \hat{a}$

Uses the phase-shift identity $\cos x = \sin(x + \pi/2)$:

  1. Load constant $\pi/2$ (quantized).
  2. Compute $a' = a + \pi/2$ via qadd.
  3. Return qsin(a').

Cost: qsin cost + 1 row.

qtan(a) — $\tan \hat{a}$

Uses the ratio identity $\tan x = \sin x / \cos x$:

  1. Compute qsin(a) and qcos(a) in sequence.
  2. Return qdiv(sin_a, cos_a).

Warning: undefined (division by zero) when $a = \pi/2 + k\pi$; the prover must avoid such inputs. Cost: ~940 rows (2× qsin + qdiv).

9.5.5 Hyperbolic Functions

qsinh(a) — $\sinh \hat{a}$

Uses the definition $\sinh x = (e^x - e^{-x}) / 2$:

  1. Compute $e^a$ via qexp(a).
  2. Compute $e^{-a}$ via qexp(neg(a)).
  3. Numerator: qsub(ea, e_na).
  4. Load constant $2_q$; return qdiv(numerator, 2).

Cost: ~540 rows (2× qexp + qdiv).

qcosh(a) — $\cosh \hat{a}$

Uses $\cosh x = (e^x + e^{-x}) / 2$. Same structure as qsinh but with qadd instead of qsub. Cost: ~540 rows.

qtanh(a) — $\tanh \hat{a}$

Uses $\tanh x = \sinh x / \cosh x$:

  1. Compute qsinh(a) and qcosh(a).
  2. Return qdiv(sinh, cosh).

Cost: ~1100 rows.

9.5.6 Power and Root

qpow(x, exponent) — $\hat{x}^{\hat{a}}$

Uses the identity $x^a = e^{a \ln x}$:

  1. Compute $\ln x$ via qlog(x) (requires $x > 0$).
  2. Compute $a \cdot \ln x$ via qmul(exponent, log_x).
  3. Return qexp(a_logx).

Cost: qlog + qmul + qexp $\approx$ 560 rows.

qsqrt(x) — $\sqrt{\hat{x}}$

Uses the identity $\sqrt{x} = x^{1/2}$:

  1. Load constant $1/2$ (quantized as $S / 2$).
  2. Return qpow(x, 0.5).

Cost: same as qpow, ~560 rows.

10. ZK Machine Learning Circuits

Each ML model is assembled from the arithmetic and gadget primitives defined in Sections 8 and 9. The common circuit pattern is:

  1. Model weights and training data go into advice columns (private witness).
  2. Public outputs (final predictions, loss value) go into instance columns — the verifier checks these directly.
  3. Every arithmetic step is enforced by a gate constraint or lookup argument.
  4. The prover fills the table and generates a proof; the verifier checks it without seeing any private data.

10.1 Linear Regression

Uses FixedPointChip<F, 32> — 32.32 fixed-point, $S = 2^{32}$.

Circuit Layout

For a $d$-dimensional model with batch size $m$, the column assignment is:

Column typeContents
Advice$w_1,\dots,w_d$, $b$ (weights/bias); per-sample $x_1^{(j)},\dots,x_d^{(j)}$; intermediate products $w_i x_i^{(j)}$; running dot-product accumulator; residuals $e^{(j)} = \hat{y}^{(j)} - y^{(j)}$; per-weight gradient accumulators $g_i = \sum_j e^{(j)} x_i^{(j)}$
InstanceUpdated weights $w_i'$ and bias $b'$ after one SGD step (public: verifier checks these)
Fixed/SelectorActivates qmul / qadd / qsub gates at the correct rows; stores the constant $\eta/m$ (learning rate pre-divided by batch size)

Inference

Given weight vector $\mathbf{w} \in \mathbb{R}^d$, bias $b$, and input $\mathbf{x} \in \mathbb{R}^d$:

$$\hat{y} = \mathbf{w} \cdot \mathbf{x} + b = \sum_{i=1}^d w_i x_i + b.$$

Circuit steps:

  1. inner_product(w, x) — $d$ calls to qmul followed by $d-1$ calls to qadd, yielding $\mathbf{w}\cdot\mathbf{x}$ (~$11d$ rows).
  2. qadd(wx, b) — adds the bias (1 row).

Total inference cost: ~$11d + 1$ rows per sample.

Training (Mini-Batch SGD)

Given a batch of $m$ samples $\{(\mathbf{x}^{(j)}, y^{(j)})\}_{j=1}^m$, one SGD step is:

$$\text{Loss} = \frac{1}{2m} \sum_{j=1}^m \bigl(\hat{y}^{(j)} - y^{(j)}\bigr)^2$$ $$\frac{\partial \text{Loss}}{\partial w_i} = \frac{1}{m} \sum_{j=1}^m e^{(j)} \cdot x_i^{(j)}, \qquad e^{(j)} = \hat{y}^{(j)} - y^{(j)}$$ $$w_i \leftarrow w_i - \frac{\eta}{m} \cdot \sum_{j=1}^m e^{(j)} x_i^{(j)}, \qquad b \leftarrow b - \frac{\eta}{m} \cdot \sum_{j=1}^m e^{(j)}$$

The circuit for train_one_batch executes the following sequence:

  1. Load learning rate. The scalar $\eta/m$ is loaded as a witness cell (the division by $m$ happens off-circuit; the circuit only sees the pre-scaled constant).
  2. Forward pass (for each sample $j = 1, \dots, m$):
    • inner_product(w, x[j]) — dot product (~$11d$ rows).
    • qadd(wx, b) — add bias (1 row).
    • qsub(y_hat, y_truth[j]) — compute residual $e^{(j)}$ (1 row).
  3. Gradient accumulation (for each weight $i = 1, \dots, d$):
    • For each sample $j$: qmul(e[j], x[j][i]) — scale residual by feature (1 row).
    • qsum([qmul results]) — accumulate over samples ($m-1$ rows).
  4. Weight update (for each weight $i$):
    • qmul(lr, g_i) — scale gradient by $\eta/m$ (1 row).
    • qsub(w[i], delta_w_i) — subtract from weight (1 row).
  5. Bias update:
    • qsum(e[]) — sum all residuals.
    • qmul(lr, partial_b) + qsub(b, delta_b).
Loss is a hint, not a constraint. The MSE loss value is computed off-circuit (using dequantization on the residuals) and printed as a warning log. It is not constrained in the circuit — the circuit only enforces that the weight update formula was applied correctly. The verifier trusts the circuit constraints, not the logged loss value.
Overflow prevention. Each qmul gate enforces $a \cdot b = c \cdot 2^P + r$ with a range check $0 \leq r < 2^P$. The range check uses a lookup argument against the table $\{0, \dots, 2^{15}-1\}$, decomposing $r$ into two 15-bit chunks.

10.2 Logistic Regression

Uses FixedPointChip<F, 63> — 63.63 fixed-point, $S = 2^{63}$. The higher precision is necessary because the sigmoid function and the $\exp$ approximation amplify rounding errors near $z = 0$.

Inference

The logistic model applies the sigmoid activation to the linear combination:

$$\hat{y} = \sigma(\mathbf{w} \cdot \mathbf{x} + b) = \frac{1}{1 + e^{-(\mathbf{w} \cdot \mathbf{x} + b)}}.$$

The exact circuit operations, in order:

  1. inner_product(w, x) — $\mathbf{w}\cdot\mathbf{x}$ (~$11d$ rows).
  2. qadd(wx, b) — logit $z = \mathbf{w}\cdot\mathbf{x} + b$ (1 row).
  3. neg(z) — negate: $z' = -z$ (1 row, field negation, no cost).
  4. qexp(z') — $e' = e^{-z}$ via $2^{z'/\ln 2}$ (~260 rows).
  5. qadd(e', 1_q) — $d' = 1 + e^{-z}$; the constant $1_q = S = 2^{63}$ is stored as a fixed-column constant (1 row).
  6. qdiv(1_q, d') — $\hat{y} = 1 / d'$ (~20 rows).

Total inference cost: ~$11d + 283$ rows per sample.

Training (Binary Cross-Entropy, Mini-Batch SGD)

The loss function is:

$$\text{BCE} = -\frac{1}{m}\sum_{j=1}^m \bigl[y^{(j)} \ln \hat{y}^{(j)} + (1 - y^{(j)}) \ln(1 - \hat{y}^{(j)})\bigr].$$

The gradient simplifies to the same form as linear regression (a consequence of the sigmoid being the canonical link for the Bernoulli distribution):

$$\frac{\partial \text{BCE}}{\partial w_i} = \frac{1}{m} \sum_{j=1}^m \bigl(\hat{y}^{(j)} - y^{(j)}\bigr) \cdot x_i^{(j)}.$$

The circuit for train_one_batch executes:

  1. Load learning rate. $\eta$ is loaded as a fixed constant (not divided by $m$ here — the caller passes a per-batch rate learning_rate_batch).
  2. Forward pass (for each sample $j$):
    • Same 6-step inference chain above, yielding $\hat{y}^{(j)}$.
    • qsub(y_hat, y_truth[j]) — residual $e^{(j)}$ (1 row).
  3. BCE loss (off-circuit hint only). For each sample, the dequantized $\hat{y}^{(j)}$ and $y^{(j)}$ are used to compute $-\ln(\hat{y}^{(j)})$ or $-\ln(1-\hat{y}^{(j)})$ in floating point. The sum is printed as a warning log. No BCE-related rows are added to the circuit.
  4. Gradient accumulation (same structure as linear regression):
    • For each weight $i$ and sample $j$: qmul(e[j], x[j][i]).
    • qsum over samples; qmul(lr, g_i); qsub(w[i], delta).
  5. Bias update — same as linear regression.

The wrapper train_multi_batch simply calls train_one_batch in a loop over batches, threading the updated weights $(w, b)$ from one call to the next. All batches land in the same proof.

Precision note: Using $P = 63$ (instead of 32) means $S = 2^{63} \approx 9.2 \times 10^{18}$. The quantization error per value is $\leq 2^{-64} \approx 5.4 \times 10^{-20}$. This is needed because $e^{-z}$ for $z$ near 0 is sensitive: a small error in the exponent produces a first-order error in $\hat{y}$, which then feeds into the gradient. With $P = 32$ the accumulated gradient error would be large enough to corrupt convergence.

10.3 Decision Tree

Uses FixedPointChip<F, 63> for both inference and training. The tree is stored as a flat array of nodes.

Node Encoding

Each node occupies 5 consecutive slots in the flat array:

$$\text{tree}[5k + 0] = \text{data\_slot}_k, \quad \text{tree}[5k+1] = \text{split}_k, \quad \text{tree}[5k+2] = \text{left\_idx}_k, \quad \text{tree}[5k+3] = \text{right\_idx}_k, \quad \text{tree}[5k+4] = \text{cls}_k.$$
  • data_slot: the feature index to compare (integer, not quantized).
  • split: the threshold value (quantized fixed-point).
  • left_idx / right_idx: child node indices. For a leaf node, $\text{left\_idx} = \text{right\_idx} = k$ (self-loop).
  • cls: the predicted class if this is a leaf (raw integer, not quantized). Set to the sentinel MASK_CLS = 255 for internal nodes.

All tree values are stored in fixed columns (compile-time constants), so they are part of the verification key and cannot be altered.

Inference Circuit

Given: the flat tree array (fixed), input feature vector $\mathbf{x}$ (advice witness), and a parameter max_path_len (the maximum depth to simulate). Steps:

  1. Initialize. Load $\mathbf{x}$ into advice cells $x_{\text{adv}}$. Set $\text{node\_idx} = 0$.
  2. Repeat for $\ell = 1, \dots, \text{max\_path\_len}$:
    1. $\text{offset} = \text{node\_idx} \times 5$ via mul(node_idx, 5).
    2. $\text{data\_slot} = \text{tree}[\text{offset}]$ via select_from_idx(tree, offset+0).
    3. Load a fresh copy of $\mathbf{x}$ and constrain each $x_{\text{copy},i} = x_{\text{adv},i}$ via copy constraints (ensures the same input is used at every step).
    4. $\text{data} = x_{\text{copy}}[\text{data\_slot}]$ via select_from_idx(x_copy, data_slot).
    5. $\text{split} = \text{tree}[\text{offset}+1]$ via select_from_idx.
    6. $\text{diff} = \text{data} - \text{split}$ via qsub; $\text{is\_less} = \texttt{is\_neg}(\text{diff})$.
    7. $\text{left} = \text{tree}[\text{offset}+2]$, $\text{right} = \text{tree}[\text{offset}+3]$ via select_from_idx.
    8. $\text{node\_idx} = \text{select}(\text{left}, \text{right}, \text{is\_less})$.
  3. Read class. After the loop: $\text{offset} = \text{node\_idx} \times 5$; $\text{cls} = \text{tree}[\text{offset}+4]$.

Because self-loops are enforced for leaf nodes, node_idx freezes at the first leaf reached and stays there for the remaining steps. The final cls is therefore always the correct prediction regardless of path length.

Cost per step: ~$5 \times O(n)$ for select_from_idx (where $n$ = tree size) + qsub + is_neg + select. Total: max_path_len $\times$ (~$5n + 18$) rows.

ZK circuit constraint: fixed computation size. ZK circuits require every execution path to have the same number of rows. A decision tree has variable-depth paths. Two techniques enforce uniformity:
  • Self-loops at leaves: once a leaf is reached, both child pointers point back to the same node, so the traversal stays put for the remaining steps without revealing when the leaf was reached.
  • Complete binary tree padding: the tree is padded to a complete binary tree of depth max_depth by adding fake internal nodes. These nodes are unreachable (since real leaves have self-loops), but their presence keeps the tree structure uniform and the fixed-column layout regular.

Training Circuit

Training builds the tree layer by layer (BFS), executing the following sub-circuits.

Gini Impurity (gini)

Given a dataset of $n$ samples, a feature slot, a split value, and a bitmask $m_i \in \{0,1\}$ indicating which samples are active in the current node, computes the weighted Gini impurity of the proposed split:

$$\text{Gini}(S) = 1 - \sum_{k=0}^{K-1} \left(\frac{n_k}{n}\right)^2, \qquad \text{Split cost} = \frac{|S_L|}{n}\cdot\text{Gini}(S_L) + \frac{|S_R|}{n}\cdot\text{Gini}(S_R).$$

Circuit steps for each group $G \in \{L, R\}$:

  1. For each sample $i$:
    • $\text{diff}_i = x_i[\text{slot}] - \text{split}$ via qsub; $\text{is\_less}_i = \texttt{is\_neg}(\text{diff}_i)$.
    • $\text{in\_left}_i = \text{is\_less}_i \cdot m_i$ via mul; $\text{in\_right}_i = (1 - \text{is\_less}_i) \cdot m_i$.
    • For each class $k$: $\text{is\_class}_{i,k} = \texttt{is\_equal}(y_i, k)$ (1 row); accumulate $n_{k,G} \mathrel{+}= \text{is\_class}_{i,k} \cdot \text{in\_G}_i$.
    • Accumulate group size: $|G| \mathrel{+}= \text{in\_G}_i$.
  2. Compute $p_{k,G} = n_{k,G} / |G|$ (scale and divide via div_mod_var). An epsilon of 1 is added to $|G|$ to prevent division by zero at empty nodes.
  3. $\text{Gini}_G = 1 - \sum_k p_{k,G}^2$. Each $p_{k,G}^2$ is computed via the internal square helper (a raw mul + div_mod rescale, ~5 rows).
  4. $\text{weight}_G = |G| / n$ via div_mod_var; multiply: $\text{weighted\_Gini}_G = \text{weight}_G \cdot \text{Gini}_G$.
  5. Sum: $\text{Split cost} = \text{weighted\_Gini}_L + \text{weighted\_Gini}_R$.
  6. Edge case: if $n \leq 1$ (leaf node), force Gini = 1 (highest possible, so this node is never chosen as the best split).
Best Split Search (get_split)

Iterates over all feature–sample pairs as candidate splits and finds the one with minimum Gini cost:

  1. Initialize $\text{best\_gini} = 1_q$ (maximum Gini), $\text{best\_slot} = 0$, $\text{best\_split} = x_0$.
  2. For each feature $f$ and each sample $i$ as the split candidate $s = x_i[f]$:
    • Intersect mask with the sample's own mask bit to derive the active mask.
    • Call gini(dataset_x, dataset_y, mask, slot=f, split=s).
    • $\text{is\_better} = \texttt{is\_less\_than}(\text{gini}, \text{best\_gini})$.
    • Update via select: $\text{best\_gini} \leftarrow \text{select}(\text{gini},\ \text{best\_gini},\ \text{is\_better})$, and similarly for slot and split.
  3. Return $(\text{best\_slot}, \text{best\_split})$.

Note: this is an $O(n \times F)$ loop over samples and features, where each iteration calls gini — the most expensive operation in the circuit. The total circuit size scales as $O(n^2 F K)$ for $K$ classes.

Majority Class (common_cls)

Finds the most frequent class among masked samples:

  1. For each class $k = 0, \dots, K-1$: count $c_k = \sum_i m_i \cdot \mathbf{1}[y_i = k]$ using is_equal + mul + add.
  2. Find the argmax via a sequential scan: maintain $(\text{max\_cnt}, \text{out\_cls})$; for each $k$, compute $\text{is\_better} = \texttt{is\_less\_than}(\text{max\_cnt}, c_k)$ and update both via select.
Full Tree Training (train)

Builds a complete binary tree of depth max_depth using BFS:

  1. Initialize queue with a single all-ones mask (all samples active at the root).
  2. For each layer $\ell = 0, \dots, \text{max\_depth}-1$: process all nodes in the current BFS level:
    1. Pop the current node's mask from the queue.
    2. Count active samples: $n_{\text{node}} = \texttt{qsum}(\text{mask})$.
    3. $\text{is\_leaf} = \texttt{is\_less\_than}(n_{\text{node}},\ \text{min\_size})$ — force leaf if too few samples.
    4. Call get_split to get $(\text{best\_slot}, \text{best\_split})$.
    5. Update left/right masks: for each sample $i$, multiply the current mask by $\text{is\_left}_i$ or $\text{is\_right}_i$. Push both to the queue for the next layer.
    6. Call common_cls on the current mask to get $\text{cls}$.
    7. Emit node record:
      • If not last layer: $\text{left\_child} = 2k+1$, $\text{right\_child} = 2k+2$ (BFS indices). If forced leaf, override both to $k$ (self-loop) and set best_slot = best_split = 0, $\text{cls} = \text{MASK\_CLS} = 255$ as sentinel.
      • If last layer: always emit a leaf: $\text{left\_child} = \text{right\_child} = k$, $\text{cls} = \texttt{common\_cls}$.
  3. Flatten all node records into the output flat array $[\text{slot}_0, \text{split}_0, \text{left}_0, \text{right}_0, \text{cls}_0, \text{slot}_1, \dots]$.

11. Putting It All Together

Here is the complete pipeline from a machine learning computation to a verified proof:

11.1 What the Proof Guarantees

A valid Halo2 proof provides four formal guarantees:

  1. Correctness: The prover executed the ML computation faithfully — every gate constraint $g_j(\omega^i) = 0$ is satisfied at every row, every range check passes, and every copy constraint is respected. The verifier confirms this without re-running the computation.
  2. Completeness: An honest prover who knows a valid witness can always produce an accepting proof. No correct computation is ever rejected.
  3. Soundness: A cheating prover cannot produce a valid proof for an incorrect computation except with probability $\leq \frac{D}{|\mathbb{F}_p|} \approx 2^{-234}$ (from the Schwartz-Zippel bound in Section 6.5). In practice this is modelled as negligible: $\leq 2^{-128}$ after accounting for all attack surfaces.
  4. Zero-Knowledge: The proof reveals nothing about the private inputs (model weights, training data, intermediate activations) beyond what is already implied by the public outputs. This holds because the witness polynomials are blinded with random values before commitment.

Together, soundness and zero-knowledge mean: the verifier is convinced the ML model was trained/inferred correctly, but learns nothing about the model itself — useful for proving model quality without disclosing proprietary weights.

  • Linear regression inference ($d=10$): ~110 rows — trivially fits.
  • Logistic regression training ($d=10$, $m=100$ samples, 1 epoch): ~$100 \times (110 + 500) \approx 61{,}000$ rows — fits comfortably.
  • Decision tree ($\text{depth}=5$, 32 nodes): ~$32 \times 10 \approx 320$ rows for comparisons — very cheap.

11.3 Limitations and Future Work

  • Circuit size: $k=18$ limits the circuit to $2^{18}$ rows. Larger models (deep neural networks, transformers) require either a larger $k$ (exponentially more prover memory) or recursive proof composition (chaining multiple proofs).
  • Fixed-point precision: Rounding errors accumulate across layers. Deep models require careful precision tuning ($P$ bits) to avoid divergence.
  • Non-polynomial operations: Operations like $\max$, $\text{argmax}$, and conditional branches require special gadgets (range decomposition, bit extraction) that are significantly more expensive than their floating-point equivalents.
  • Training vs. inference: Proving a full training loop is far more expensive than proving a single inference. For large models, it is more practical to prove inference only (with public weights) and trust the training process, or use recursive proofs to batch training steps.
